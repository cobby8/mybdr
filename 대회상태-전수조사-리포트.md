> 작성: 2026-06-14 (Cowork) · 목적: "종료 대회가 모집 중으로 노출" 등 상태 오표시 원인 전수조사
> 범위: **코드 전수조사 완료** / **DB 실측은 Claude CLI 핸드오프** (Cowork 샌드박스는 운영 Supabase 접근 차단 — allowlist 외)

## 0. 한 줄 결론

대회 상태는 **DB `status` 문자열 하나만** 보고 표시된다. **날짜(end_date)를 전혀 보지 않는다.**
그래서 운영자가 status 를 `completed` 로 바꾸는 걸 누락하면, 종료일이 한참 지나도 영구히 "모집 중"으로 뜬다.
자동 종료 장치는 있으나 "**모든 매치가 100% 기록 완료**"일 때만 작동 → 매치를 안 쓰는 대회는 절대 자동 종료 안 됨.

## 2. 근본 원인 (코드 확정 — 5건)

**[원인 1] 날짜 미반영 (핵심)**
`deriveV2Status` 와 `TOURNAMENT_STATUS_LABEL` 둘 다 `status` 문자열만 매핑. `end_date` 경과 여부를 보지 않음.
→ status 가 `registration`/`active`/`published` 인 채로 남으면 종료일이 지나도 영구 "모집 중".
이것이 스크린샷에서 종료 대회가 "모집 중"으로 뜨는 **직접 원인**.

**[원인 2] 자동 종료의 한계**
`auto-complete.ts` 가 유일한 자동 종료 경로인데 조건이 "**모든 매치가 종료 상태**":
- 매치 0건 대회(매치 미생성·외부기록·단순등록) → `no-op` (`auto-complete.ts:89`). 자동 종료 영구 불가.
- knockout format 결선 미생성 시 보류.
- 매치를 끝까지 기록 안 하면(`finished < total`) 종료 안 됨.
→ **날짜 기반 종료 트리거가 아예 없음.** 매치를 안 쓰는 운영 패턴이면 status 가 영원히 안 바뀜.

**[원인 3] 마감임박 배지 사문화**
`deriveV2Status` 의 "마감임박(7일 이내)" 계산이 `t.registration_end_at` 에 의존하는데, 목록 API 응답에 그 필드가 **없음**(`route.ts` 직렬화·`TOURNAMENT_LIST_SELECT` 미포함) → 항상 `undefined`.
→ 날짜로는 마감임박이 절대 안 뜨고, status 가 정확히 `registration_closed` 일 때만 뜸.

**[원인 4] 이중 매핑 소스**
목록 = `deriveV2Status`(5상태) / 상세·홈·admin = `TOURNAMENT_STATUS_LABEL`(4상태). 현재 키 집합은 일치하나, 한 쪽만 갱신되면 목록 vs 상세 라벨 불일치 위험. 단일 소스 아님.

**[원인 5] 알 수 없는 status 폴백 불일치**
`deriveV2Status` 는 미지의 status 를 "접수예정"으로 폴백 후 **숨김**. `TOURNAMENT_STATUS_LABEL` 은 raw status 를 **그대로 노출**. DB 에 매핑 외 값이 있으면 목록엔 안 보이는데 상세엔 보임. (실측 [D] 필요)

## 3. DB 실측 결과 (Claude CLI 실행 완료 — 2026-06-14, 전체 61건)

> SELECT 전용 감사 (운영 영향 0). Cowork 샌드박스는 Supabase 도메인 차단으로 직접 실행 불가 → CLI 에서 실행됨.

**status 분포:** `published 53` · `completed 5` · `in_progress 1` · `cancelled 1` · `draft 1`

**5분류 집계:**

| 분류 | 건수 | 내용 |
|---|---|---|
| **[A]** 모집(published)인데 종료일 경과 | **50** | 핵심 오표시군. 아래 47+3 으로 분해 |
| **[B]** 진행중인데 경과 | **1** | 열혈농구단 SEASON2 (in_progress, 종료일 2024-06-27) |
| **[C]** 종료인데 시작 미래 | 0 | 없음 |
| **[D]** 미분류 status | 0 | ✅ 정체불명 값 없음 (전부 알려진 값) |
| **[E]** 경과 + 매치 0 + 미종료 | **47** | 공지전용 "껍데기" 대회 (cafe import) |

**[A] 50건의 분해:**
- **47건 = 공지전용 껍데기** ([E]와 동일군). 매치 0·팀 0. 우리 시스템으로 경기를 안 한, 공지만 올린 대회 → 매치 기반 자동종료가 **영영 안 돎**.
- **3건 = 실제 진행됐는데 종료 처리 누락** (매치 7/팀 6, status=published):
  2026 4차 BDR 뉴비리그 · 2026 5차 BDR 뉴비리그 · 2026 6차 BDR 스타터스리그.
  (5차는 순위/결승까지 정리됐으나 status 는 여전히 published)

**핵심 인사이트:** "우리 시스템 안 쓰고 **공지만 올리는 대회**(매치 0)"는 앞으로도 계속 생기는 **정상 패턴**. → 이 군은 매치가 아니라 **날짜**로 종료 판정해야 한다(= 원인 1·2 의 근본 처방이 날짜 기반 표시/정합임을 실증).

## 4. 수정 방향 — 결정 완료 (3-Phase)

수빈 결정(2026-06-14): **① 표시 레이어 먼저(무위험) · ② 공지전용은 날짜 지나면 완료 처리.**

| Phase | 성격 | 내용 | 대상 | 프롬프트 |
|---|---|---|---|---|
| **1** | 코드 (DB 0) | `effectiveTournamentStatus(status, start, end)` 공통 함수 — 종료군/준비중 외 **종료일 경과 시 '종료' 표시**. **공개 화면만**, admin 은 raw 유지. 원인 1·4·5 해소 + 재발 영구 방지 | 50건 즉시 정상화 + 향후 전부 | `대회상태-Phase1-CLI프롬프트.md` |
| **2** | DB UPDATE (가드) | 실진행 3건(뉴비4·5·스타터6) → `completed` + **우승팀 수동 확정**(champion_team_id 자동 입력 코드 없음). 열혈은 결선 종료 후 별도 | 3건 (+열혈 보류) | `대회상태-Phase2-CLI프롬프트.md` |

**설계 원칙(P2·P3 공통):** 매치>0 = 경기 기반 종료 / **매치=0 = 날짜 기반 종료**. 이 분리로 열혈·진짜 리그가 날짜 강제종료에서 자동 제외 → 오작동 0.
