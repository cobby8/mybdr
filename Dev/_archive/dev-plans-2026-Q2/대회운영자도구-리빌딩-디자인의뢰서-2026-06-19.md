# BDR 시안 의뢰 — 관리자 디자인시스템 신설 + 대회 모듈 전면 리빌딩

> 대상: Claude.ai(디자인 클로드) · 2026-06-19 (rev3) · 작성: Cowork(코드 실측 + 현행 시스템 분석)
> 핵심: ① **mybdr 관리자 전체에 Toss 스타일 디자인시스템 신설(프론트와 독립) — 전면 개편** ② 대회 모듈(참가신청+운영)을 그 첫 적용으로 리빌딩.
> **레퍼런스 = 현재 운영 중인 참가신청 시스템 BDR-join-v1** (repo github.com/cobby8/BDR-join-v1 · live bdr-join-v1.vercel.app). 이 UI를 **그대로 차용/재현**하는 게 이번 의뢰의 핵심.
> 1차 초점: 종별 프리셋(=종별 마스터) 복원 + 자동화 / 범위: 참가신청서 + 운영 전 화면.

## 1. ⭐ 디자인 방향 — Toss 스타일을 mybdr "관리자 디자인시스템"으로 신설, 관리자 전체 전면 개편 (사용자 확정)

**BDR-join-v1의 Toss 스타일을 mybdr 관리자 전용 디자인시스템으로 그대로 채용 + 독립.** mybdr 프론트(사용자 사이트: BDR Red·다크 기본·4px·Material Symbols·var토큰)와 **완전히 분리된 별개 디자인시스템**으로 운영한다.
- **적용 범위 = 모든 관리자/운영 페이지 (E등급 자체 디자인 영역 전체)**: `(admin)/admin/*`(슈퍼 관리자 콘솔 — Games/Teams/Users/Payments/Notifications/Plans 등), `(admin)/tournament-admin/*`(대회 운영), `(web)/partner-admin/*`(파트너), `(referee)/referee/*`(심판). → 이번 대회 모듈을 **앵커(첫 적용)**로 하고, 동일 Toss 시스템을 **관리자/파트너/심판 전 화면에 확장 적용해 전면 개편**(사용자 확정 2026-06-19).
- 메인 13룰(var토큰·다크·Material·lucide 금지·4px)은 **사용자 프론트 한정** → 관리자·파트너·심판은 본 Toss 시스템을 따른다(별개 토큰 세트).
- referee는 자체 RefereeShell, partner-admin은 자체 셸 → 셸 골격은 유지하되 **Toss 토큰·컴포넌트로 재스킨**.

**관리자 디자인시스템 토큰 (BDR-join-v1 globals.css 기준)**:
- 테마: **라이트** (배경 `#F9FAFB`, 카드 `#FFFFFF`, 본문 `#0f172a` 슬레이트).
- Primary: **Toss Blue `#3182F6`** (primary-foreground 흰색).
- Destructive: `#ef4444` 계열 / Border: `#e5e7eb` (gray-200) / Muted text: slate-500.
- **라운딩: 24px(`--radius: 1.5rem`, rounded-3xl 룩)** — 카드·버튼·모달 큰 라운딩.
- 폰트: **Pretendard** (mybdr와 동일 — CDN 동일).
- 아이콘: **lucide-react** (이 모듈 한정 허용 — BDR-join-v1 그대로). Material Symbols 강제 아님.
- 여백 넉넉 / 카드 중심 / 부드러운 그림자 (Toss 톤).

→ 디자인 클로드는 BDR-join-v1 라이브·repo의 화면을 시각 기준으로 삼아 동일 룩으로 재현. 색/라운딩/여백/컴포넌트 형태를 픽셀 수준으로.

## 2. 배경 — 왜 리빌딩 (코드 실측)
- **종별 프리셋 소실**: 사용자가 만든 "종별 프리셋"의 실체 = BDR-join-v1의 **`admin_categories` 종별 마스터**(CategoryManager 관리). mybdr 포팅 시 `DivisionGeneratorModal`만 가져오고 마스터 테이블·UI 미이식 → 매번 수동. (참고: `tournament_presets` 테이블/PresetManager는 BDR-join-v1에서도 **미연결 고아**였음 → 베끼지 말 것.)
- **IA 복잡**: mybdr는 위저드→SetupHub 8~9카드→분산 라우트. BDR-join-v1은 사이드바 4메뉴 + 대회상세 통합으로 **훨씬 단순** → 이 단순 구조를 채택.

## 3. 참가신청서 (사용자용) — ⭐ 기존 팀원 선택 기반 (명단 직접입력 폐지)

> 핵심 차이: **mybdr는 팀에 이미 가입 선수가 있다.** BDR-join-v1처럼 명단을 일괄 붙여넣기/직접 입력할 필요 없음 → **기존 팀 로스터에서 출전 선수를 선택**하는 방식으로 전환.

**3단계 위저드**(Toss 룩) — BDR-join-v1 흐름 차용하되 **유니폼 단계 폐지**(팀 페이지에서 이미 설정) + **명단 입력 → 기존 팀원 선택**으로 교체:
- **Step 1 참가팀 선택 + 정보 확인 (필수 첫 단계)**: 가입 팀이 여러 개일 수 있으므로 **맨 처음 참가할 팀을 선택**. 선택 즉시 **팀 정보 확인**(팀명 한/영, 대표자, 연락처, 지역, 로스터 인원) 표시. 로그인 세션 기반(비번 입력 불필요). 팀 1개면 자동 선택 + 확인만.
- **Step 2 종별·디비전 선택**: 종별 버튼 → 디비전 버튼. **각 디비전 정원 표시(현재/정원)**, 초과 시 **"대기접수" 뱃지(주황 강조)**. (div_caps + 집계)
- **Step 3 출전 선수 선택 (명단 입력 X)**: 선택한 팀의 **로스터(기존 가입 선수)에서 체크박스로 출전 선수 선택**. 선수 정보(이름/등번호/포지션/생년/선출)는 **이미 보유 → 표시만**, 신규 입력 없음. ⏳ **출전 최소 인원 + 게스트(팀원 외) 추가 허용 = 추후 결정**(시안은 최소인원 가드·게스트 추가 UI 자리를 비워두되 토글 가능하게 제안).
- **유니폼 입력 없음**: 팀 페이지에서 관리자가 설정한 팀 유니폼을 그대로 사용(표시만, 신청 시 미설정).
- **완료 화면**: 입금 계좌(은행/계좌/예금주/참가비) + 복사. 정원초과면 "대기 접수하기".
- 상태: 제출 시 서버 정원 재판정 → APPLIED / WAITING. 조회·수정 = 마이페이지 내 신청 내역에서.

## 4. 종별 마스터 + 프리셋 복원 (P0 핵심)

**4-1. 종별 마스터 관리(admin_categories 패턴)** — 잃어버린 "종별 프리셋"의 복원체.
- 관리자 설정의 "종별 관리" 화면. 종별 CRUD + `divisions`·`ages` **태그 입력**(Enter 추가/X 제거).
- **시드 4종(사용자 실제 데이터 그대로 복원)**:
  - 일반부: D3, D4, D5, D6, D7, D8
  - 유청소년: 하모니, i1, i2, i3, i4 / 연령 U8~U18
  - 대학부: U1, U2, U3
  - 시니어: S1, S2, S3 / 연령 +40,+45,+50,+55,+60,+65,+70

**4-2. 대회 생성 시 자동화**:
- `DivisionGeneratorModal`이 종별 마스터를 읽어 종별을 찍어냄(성별→종별→디비전 선택).
- 종별 확정 시 **`TournamentDivisionRule` 자동 생성**(코드/라벨/format/settings/정원/참가비). 수동 재입력 제거.
- **이전 대회 복사**(TournamentCopyModal 패턴) — format/settings 포함 통째 프리필.

## 5. 관리자 도구 — BDR-join-v1 UI 그대로 차용

**5-1. IA (사이드바 4메뉴)** — BDR-join-v1 구조 채택:
- 대시보드 / 대회 관리 / 참가팀 관리 / 설정(종별 마스터·대표자).
- mybdr의 SetupHub 8~9카드 난립을 **이 단순 구조로 대체**.

**5-2. 대회 상세 = 참가팀 관리 + 대진표 통합 1화면**:
- 참가팀: 종별/디비전/상태 필터, **인라인 상태 편집**, **입금완료(paid) → 자동 CONFIRMED**, 대기팀 자리 발생 시 알림.
- 상태 머신: APPLIED / WAITING / CONFIRMED / CANCELED → mybdr `TournamentTeam.status`(pending/approved/rejected/paid)에 매핑.
- 대진표: 기존 BracketManager 패턴(듀얼/리그/토너먼트).

**5-3. 대회 생성 폼(TournamentForm 패턴)**: 3컬럼 단일 폼 — 기본정보+결제 / 경기장+상세(주최·주관·후원·경기시간·경기구·경기방식)+미디어 / 종별·디비전 구성(DivisionGeneratorModal). "이전 대회 불러오기" 헤더 버튼.

## 6. 데이터 정합 & 백엔드 후속 (시안 외 — Cowork/CLI)
- **호환**: `Tournament.categories(=divs)/div_caps/div_fees/places/entry_fee/bank_*/gender/gameTime/gameBall/gameMethod`, `TournamentTeam.status/payment_status`, players(이미 보유 — 선택만).
- **추가 필요**: `admin_categories` 마스터 테이블 + 4종 시드 / 종별 확정 시 `TournamentDivisionRule` 자동 생성 / 정원 집계 WAITING 판정 / 참가신청 = **팀 로스터 조인**(명단 입력 폐지 핵심).
- **옛 종별 프리셋 데이터 복원 조사**: 사용자가 BDR-join-v1(운영 Supabase)에 만든 admin_categories가 살아있음(시드 확인) → mybdr로 마이그레이션 가능.

## 7. 산출물
- **관리자 디자인시스템 정의** (Toss 토큰·컴포넌트 키트: 버튼/카드/입력/뱃지/모달/사이드바/테이블 — 관리자 전 화면 공통). 이게 1순위 기반물.
- 대회 모듈 시안(위 시스템 적용 — 첫 적용):
  - **참가신청서 3단계**(사용자용: 참가팀 선택·확인 → 종별·디비전 → 출전선수 선택. 유니폼 단계 없음)
  - **종별 마스터 관리**(CategoryManager) + 4종 시드
  - **관리자 IA**(사이드바 4) + **대회 상세**(참가팀+대진표) + **대회 생성 폼** + **대시보드**
- (확장) 동일 시스템을 `(admin)/admin/*` 슈퍼관리자 콘솔 + `partner-admin/*` + `referee/*` 전 화면에 적용 — 후속 배치(셸 골격 유지·Toss 재스킨).
- 전달: zip → Cowork가 `BDR-current/`에 별도 핸드오프 패키지로 동기화. README + **DATA-BINDING**(admin_categories·DivisionRule 자동생성·로스터 조인·WAITING 판정 계약).
- **단계 분할 권장**: ① **관리자 디자인시스템 + 참가신청서 + 종별 마스터**(P0) → ② 관리자 IA·대회상세·생성폼 → ③ 슈퍼관리자 콘솔 + partner-admin + referee 전체 확장.
