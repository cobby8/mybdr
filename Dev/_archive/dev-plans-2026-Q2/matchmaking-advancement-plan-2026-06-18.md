# 픽업/게스트 경기 매칭 고도화 기획서

> 작성: 2026-06-18 (Cowork 허브) · 범위: **픽업(game_type=0) / 게스트(game_type=1) 경기 중심**
> 목표: 유저가 원하는 경기를 쉽게 **찾고 → 신청하고 → 실제 참가**하고, 주최자는 쉽게 **개최·관리·마감·확정**하고, 끝나면 **서로 평점**을 매기는 end-to-end 루프를 실사용 가능한 수준으로 완성.
> source of truth: 본 문서는 **실측(코드 grep) 기반** baseline. 시안 가정이 아니라 운영 src/ 코드 근거.

## 0. 핵심 매칭 루프 (이 문서가 다루는 전체 흐름)

```
[찾기] 유저가 경기 검색/필터
   ↓
[신청] 일반/게스트 신청 (대기 status=0)
   ↓
[승인] 주최자 승인 (status=1) → current_participants++
   ↓
[마감/확정] 정원 도달 시 모집중(1)→확정(2)
   ↓
[리마인드] 경기 전 알림
   ↓
[경기 당일] 출석 확정 (attended_at)
   ↓
[종료] status→완료(3)
   ↓
[평점] 리포트 작성 (전체평점 + MVP + 동행자 평점/신고)
   ↓
[반영] 매너점수/MVP가 프로필에 누적 → 다음 매칭 신뢰 신호
```

**현재 이 루프는 양 끝(찾기·신청, 평점 입력)은 구현돼 있으나, 가운데 "성사·확정·출석" 연결고리가 비어 있어 실사용 시 끊긴다.** 아래가 그 진단.

## 1. 현황 점검 (4개 영역 · 실측 기반)

### 1-A. 경기 찾기 / 검색 / 필터 — **대체로 완성, UX 보강 여지**

| 페이지네이션 | ✅ (take=60) | — |

### 1-B. 참가 신청 → 실제 참가 — **신청/승인은 완성, "성사·출석"이 비어 있음**

| 항목 | 상태 | 근거 |
|------|------|------|
| 일반 신청 | ✅ | `POST /api/web/games/[id]/apply` |

| 신청 취소 | ✅ | `DELETE .../apply/cancel` |
| 신청 알림 (생성/승인/거절) | ✅ | NOTIFICATION push |

| 항목 | 상태 | 근거 |

## 2. 핵심 갭 & 우선순위

루프가 "실제로 성사·완결"되려면 **가운데 연결고리(성사→출석→평점 반영)** 가 우선. 찾기 UX는 유입 단계라 그 다음. 우선순위:

| 우선 | 갭 | 왜 치명적인가 | 영역 |
|:---:|------|------------|------|
| **P0** | 정원 마감 자동 전환 + 취소 status 버그(4 vs 5) | 경기가 "성사" 표시되지 않음. 모집중인데 자리 없음 = 신뢰 붕괴 | 주최/참가 |
| **P0** | 경기 취소/변경 시 신청자 알림 | 주최자가 취소해도 신청자가 모름 = 노쇼·헛걸음 | 주최 |
| **P1** | 대기열(waitlist) | 정원 차면 신청 자체 거절 → 취소 발생 시 빈자리 안 채워짐 | 참가 |
| **P1** | 출석 확정(attended_at) → 평점 권한 연동 | 노쇼한 사람도 평가 가능 = 평점 신뢰 저하 | 참가/평점 |
| **P1** | 경기 종료 후 평점 작성 유도(알림+배너) | 평점 시스템은 완성됐는데 아무도 안 들어옴 | 평점 |
| **P2** | 평점/매너점수 프로필 노출 | 평가의 보상·동기. 안 보이면 평가할 이유 없음 | 평점 |
| **P2** | 찾기 UX (정렬·빠른필터·임박 표시·빈상태) | 유입·전환율. 기존 동작 위 개선 | 찾기 |
| **P3** | 참가비 결제 | 유료 경기 한정. 외부 PG 연동 필요 = 비용/난이도 큼 | 참가 |

> **P3 결제는 본 로드맵에서 "설계만" 하고 구현은 분리 권장** (외부 PG·정산·환불 정책 = 별도 의사결정).

## 3. 단계별 로드맵 (Phase M1 ~ M6)

> 명명: **M = Matchmaking 고도화 phase**. 기존 Phase 번호와 충돌 방지용 독립 네임스페이스. (phase-ledger.md 등재 시 수빈 확정)
> 모든 단계 공통 가드: `subin` 브랜치 작업 → `subin→dev` PR. **운영 DB 단일** → schema 변경은 NULL 허용 ADD COLUMN 같은 무중단만 자동, 그 외 수빈 검토. destructive 0.

### Phase M1 — 성사 코어 + 취소 버그 (P0) ⭐ 최우선
**목표**: 경기가 제대로 "확정"되고, 취소가 정상 표시·통지되게.
- 정원 도달 시 `status` 1(모집중)→2(확정) 자동 전환 (승인 API + 신청 reserve 트랜잭션 내).
- 정원 미달로 떨어지면(취소/거절 발생) 2→1 복귀.
- 취소 status 통일: DELETE의 `status=5`를 **4(취소)로 정정**하거나 `STATUS_LABEL[5]` 추가 — 둘 중 하나로 일관화(코드 전수 확인).
- 취소 시 신청자 전원에게 `GAME_CANCELLED` 알림 발송 (이미 상수 존재, 호출만 연결).
- **DB 변경 없음** (status 컬럼 활용). 회귀 위험 낮음.
- 검증: 정원 채우기 → status 2 확인 / 1명 취소 → status 1 복귀 / 경기 취소 → 신청자 알림 수신 + 라벨 정상.

### Phase M2 — 대기열(Waitlist) (P1)
**목표**: 정원 차도 "대기 신청" 가능, 빈자리 발생 시 순번 안내.
- `game_applications.status`에 대기열 상태 추가 (예: 3=대기). 또는 별도 컬럼 `waitlist_position`.
- 정원 초과 신청 → 거절 대신 대기 등록.
- 승인된 참가자 취소 시 → 대기 1번에게 알림 ("자리 났어요, 신청 확정하시겠어요?") — 자동승격 vs 알림후수동 = **수빈 결정 필요**.
- **DB 변경**: ADD COLUMN `waitlist_position INT NULL` (무중단). 
- 검증: 정원 초과 신청 → 대기 등록 / 1명 취소 → 대기 승격 알림.

### Phase M3 — 출석 확정 → 평점 권한 연동 (P1)
**목표**: 실제 온 사람만 평가받고/평가하게 → 평점 신뢰.
- 주최자용 **출석 체크 UI** (경기 상세, status=2/완료 시 참가자 리스트에 출석 토글) → `attended_at` set.
- `canReportGame()` 보강: "승인됨"뿐 아니라 출석 데이터 있으면 출석자 기준으로 평가 대상 좁힘 (노쇼는 평가 가능, 노쇼는 신고 플래그로 처리하는 현 구조와 정합).
- 노쇼 자동 반영: 출석 안 한 승인자 → 매너점수에 노쇼 신호.
- **DB 변경 없음** (attended_at 기존 컬럼).
- 검증: 출석 토글 → attended_at 기록 / 리포트 작성 시 출석자/노쇼 구분 표시.

### Phase M4 — 평점 작성 유도 + 프로필 반영 (P1→P2)
**목표**: 완성된 평점 시스템에 사람을 들여보내고, 결과를 보여줘 동기 부여.
- 경기 종료(status→3) 시점에 호스트·참가자에게 **"평점 남기기" 알림** + 경기 상세/마이페이지 **배너 CTA**.
- 미작성자 리마인드 (1회, 종료 +24h).
- 프로필에 **매너점수·MVP 횟수·참여 경기 수** 노출 (manner_score 표시 UI 확인 후 보강/신설).
- 검증: 경기 종료 → 알림 수신 → CTA 진입 / 프로필에 매너점수 표시.

### Phase M5 — 찾기/검색 UX 고도화 (P2)
**목표**: 유입·전환. 기존 listGames 위에 UX 레이어.
- 정렬 옵션: **임박순(scheduled_at)·인원 임박순·최신순·가까운순(좌표 있을 때)**.
- 빠른 필터 칩: **오늘 / 이번 주말 / 내 동네 / 모집 임박(정원 80%+) / 무료**.
- 카드에 **모집 진행률 바**(current/max) + "마감임박"·"확정" 뱃지 노출.
- 결과 0건 빈 상태 + 추천(맞춤필터 끄기 / 인접 지역 제안).
- 요일/시간대 필터 실제 동작 확인·수정.
- **DB 변경 없음** (정렬/필터는 쿼리·메모리). 좌표 정렬은 court 좌표 유무 확인.
- 검증: 각 정렬·칩 동작 / 진행률 바 정확 / 0건 빈상태.

### Phase M6 — 참가비 결제 (P3, 설계만)
**목표**: 유료 경기 정산. **구현은 별도 의사결정 후.**
- 설계 산출물만: PG 후보(토스/카카오페이/네이버페이), 정산·환불·노쇼 정책, `payment_*` 필드 활용안, 기존 구독 결제(`/api/web/payments/*`) 재사용 가능성 분석.
- 본 단계는 기획서 부록으로 분리, CLI 박제 보류.

### 의존성 그래프
```
M1 (성사코어) ──┬─→ M2 (대기열)
                └─→ M3 (출석) ──→ M4 (평점유도/반영)
M5 (찾기UX) ── 독립 (M1 status 정합 후 진행 권장)
M6 (결제) ── 독립, 설계만
```

## 4. 영역별 상세 설계 노트

### 4-1. status 전환 규칙 (M1 핵심 — 전 영역 영향)
- 단일 정의 소스 = `src/lib/constants/game-status.ts`. 정수 의미를 코드 주석으로 명문화:
  `0=초안, 1=모집중, 2=확정(정원마감), 3=완료, 4=취소`.
- **status 5는 폐기** → 취소는 4로 통일. (현 DELETE 버그 정정. 기존 status=5 데이터 있으면 마이그레이션 SELECT로 건수 확인 후 UPDATE — destructive 가드 적용)
- 전환 트리거:
  - 승인 시 `current == max` → 2
  - 취소/거절로 `current < max` 이고 현재 2 → 1
  - `scheduled_at < now` → 표시상 3 (현 동적 오버라이드 유지). 실제 DB 전환은 별도 배치 or 종료처리 액션에서.
- ⚠️ 동적 오버라이드(표시 3)와 실제 DB status 불일치는 평점 권한(`canReportGame`이 status=3 요구)에 영향. **종료된 경기의 DB status를 실제 3으로 만드는 경로 필요** (M3/M4 진입 전제). → "경기 종료 처리" 액션 또는 조회 시 lazy update 검토.

### 4-2. 대기열 정책 (M2 — 수빈 결정 필요)
- 자동 승격 vs 알림 후 수동 확정.
- 대기 표시: "대기 3번" 카드 뱃지.
