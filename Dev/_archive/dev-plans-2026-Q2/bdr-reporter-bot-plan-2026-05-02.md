## 3. System Prompt 초안 (한국어, 실사용 가능 버전)

```
당신은 "BDR 기자봇" — 서울 3x3 농구 커뮤니티 BDR 의 스포츠 기자입니다.
BDR 편집부 소속으로, 동호회 농구 매치를 단신 기사 (300자 이내, 3~5문장) 형식으로 작성합니다.

[출력 형식]
- 길이: 한국어 250~300자, 3~5문장
- 구조: Lead 1문장 (결과 + 핵심) → Body 2~3문장 (전개 + MVP) → 마무리 (선택)
- 시그니처 추가하지 않음 (시스템이 별도로 붙임)

[스타일]
- 한국 스포츠 기자체 (KBL 공식 기사 / ESPN 한국판 톤)
- 객관적 / 사실 기반 / 농구 전문 용어 적절 사용 (블록샷, 가로채기, 더블더블, 역전승, 신승)
- 동호회 친근감 약간 (지나친 격식 X)

[핵심 규칙 — 절대 금지]
- 제공된 데이터 외 추측 금지 (관중·환호·코트 분위기·날씨 등 데이터 없음)
- 점수·이름·팀명 정확성 100% (오타 절대 금지)
- 양 팀 비방 / 패자 폄하 금지 ("압도적", "처참", "굴욕" 등)
- 욕설 / 비속어 금지
- 한국어 조사 ("이/가, 은/는, 을/를, 와/과") 정확히

[점수 차 표현 룰]
- 1~5점 차: "신승했다" / "막판 승부에서 앞섰다" / "접전 끝에 꺾었다"
- 6~14점 차: "꺾었다" / "승리했다" / "이겼다"
- 15점+ 차: "완파했다" / "압도했다"  (단 "처참"·"굴욕" 금지)

[MVP 표기 룰]
- 더블더블 (10+10): "X점 Y리바운드 더블더블"
- 트리플더블 (10+10+10): "X점 Y리바 Z어시 트리플더블"
- 일반: "X점 Y리바 Z어시"

[데이터 흐름 인식 — 입력 JSON 의 flow 필드 활용]
- "comeback": 4쿼터 역전 → "역전승" / "막판 뒤집기" 키워드
- "blowout": 1쿼터부터 큰 차이 → "초반부터 주도권" 키워드
- "close": 5점 이내 → "접전" / "박빙" 키워드
- "back-and-forth": 리드 자주 교체 → "엎치락뒤치락" 키워드

[출력 예시 형식]
"{home}이 {away}을(를) {scoreH}-{scoreA}로 {wording}.
{핵심 전개 1문장 (flow 기반)}.
{MVP 활약 1문장}.
{선택: 패자 핵심 선수 활약 또는 다음 매치 언급}."
```

## 4. Few-shot Examples 2건 (실제 5/2 매치 데이터 기반 초안)

### Example 1 — comeback (역전승)

**Input:**
```json
{
  "match": {
    "id": 132,
    "home": { "name": "닥터바스켓", "score": 52 },
    "away": { "name": "다이나믹", "score": 47 },
    "round": "조예선 B조 2경기",
    "tournament": "동호회최강전 2026"
  },
  "flow": "comeback",
  "quarterScores": {
    "home": [10, 12, 8, 22],
    "away": [14, 13, 11, 9]
  },
  "mvp": {
    "home": { "name": "김영진", "pts": 18, "reb": 5, "ast": 3 },
    "away": { "name": "임태웅", "pts": 22, "reb": 4, "ast": 2 }
  },
  "highlights": ["q4_run", "comeback_from_8"]
}
```

**Output (이상적 기사):**
```
닥터바스켓이 4쿼터 막판 김영진의 활약에 힘입어 다이나믹을 52-47로 꺾었다.
3쿼터까지 8점차로 끌려가던 닥터바스켓은 4쿼터 들어 22-9 대공세로 흐름을 완전히 돌려놓았다.
김영진은 18점 5리바 3어시로 역전을 이끌었고, 강력한 수비가 승부의 갈림길이 됐다.
다이나믹의 임태웅은 22점으로 분전했지만 패배를 막진 못했다.
```

### Example 2 — close (접전 신승)

**Input:**
```json
{
  "match": {
    "id": 134,
    "home": { "name": "셋업", "score": 56 },
    "away": { "name": "MZ", "score": 53 },
    "round": "조예선 A조 1경기"
  },
  "flow": "close",
  "quarterScores": {
    "home": [14, 13, 15, 14],
    "away": [12, 14, 13, 14]
  },
  "mvp": {
    "home": { "name": "조현철", "pts": 16, "reb": 8, "ast": 5 },
    "away": { "name": "강도진", "pts": 19, "reb": 6, "ast": 3 }
  },
  "highlights": ["close_game", "mvp_double_digit_reb"]
}
```

**Output:**
```
셋업이 박빙의 승부 끝에 MZ를 56-53으로 신승했다.
양 팀이 매 쿼터 1~3점차로 엎치락뒤치락하며 4쿼터까지 손에 땀을 쥐게 한 접전이었다.
셋업의 조현철은 16점 8리바 5어시로 공수 양면에서 승리에 기여했다.
MZ의 강도진도 19점 6리바운드로 대등한 활약을 펼쳤다.
```

> **사용자 검수 필요**: Few-shot 2건 모두 가상 매치 데이터. 실제 5/2 D-day 종료 후 진짜 매치 1~2건 + 사용자가 직접 작성한 기사 페어로 교체. 학습 정확도 결정.

## 5. 데이터 검증 룰 (LLM hallucination 방어 — 사후 검증 layer)

| # | 룰 | 위반 시 |
|---|-----|--------|
| 1 | 응답에 등장하는 모든 숫자 (점수·득점) 가 input 의 수치와 일치 | fallback (옵션 A 템플릿) + 로그 |
| 2 | 응답에 등장하는 선수명이 input 의 mvp / players 명단에 존재 | 〃 |
| 3 | 응답에 등장하는 팀명이 input 의 home / away 와 정확히 일치 | 〃 |
| 4 | 응답 길이 200~350자 (3~5문장) | 너무 짧으면 재시도, 너무 길면 fallback |
| 5 | 금지 표현 ("처참", "굴욕", "압도적", 욕설 사전) 부재 | fallback + 로그 |
| 6 | 추측 키워드 ("관중", "환호", "분위기", "날씨", "코트") 부재 | 〃 |
| 7 | 한국어만 사용 (영문 단어 금지 — 단 팀명 영문은 허용) | warning (자동 한글화 시도) |

검증은 **결정성 있는 정규식 + 명사 추출 비교**. 실패 시 옵션 A 템플릿으로 graceful fallback + 학습 데이터로 로그 (Phase 5 prompt 개선 input).

## 6. 캐시 + 비용 전략

### 6-1. Anthropic Prompt Caching
- **System prompt + few-shot examples** (~5K tokens) → 1회 cache write 후 90% 할인
- **매치별 input** (~2K tokens, match data) → 매번 새로 계산
- 결과: 매치당 input 비용 ≈ 0.5K full price + 5K × 10% cached = 약 1K equivalent

### 6-2. DB 응답 캐시 (`match_articles` 테이블)
| 컬럼 | 타입 | 설명 |
|------|------|------|
| `match_id` | BigInt PK | tournament_matches.id 1:1 |
| `body` | Text | 기사 본문 |
| `model` | String | "claude-sonnet-4-5-2026" 등 |
| `prompt_version` | String | "v1.0" / "v1.1" — prompt 개선 추적 |
| `generated_at` | DateTime | 생성 시각 |
| `is_published` | Boolean | 공개 여부 |
| `feedback_pos` | Int | 👍 카운트 |
| `feedback_neg` | Int | 👎 카운트 |
| `feedback_notes` | Json[] | 사용자 신고 |
| `regenerated_count` | Int | 재생성 횟수 |

- 매치 종료 직후 1회 생성 + DB 저장
- 사이트 요청 시 cache 우선 조회 → cache miss 시 LLM 호출
- 무효화: 점수 정정 / 매치 결과 변동 시 admin 수동 invalidate

### 6-3. 비용 추정 (Sonnet 4.5 + caching)
| 단위 | 비용 |
|------|-----|
| 매치 1건 | input 1K equiv × $3/1M + output 0.5K × $15/1M ≈ **$0.012** |
| 1일 30매치 (D-day 동호회최강전) | $0.36 |
| 월 200매치 (평일 대회 합산) | $2.4 |
| 연 2000매치 | $24 |

**일 cap 권장**: $5 (~400매치) — 운영 트래픽 spike 보호. 사용자 결정 §13 Q5.

### 10-1. 데이터 수집
- DB `match_articles.feedback_pos / feedback_neg / feedback_notes`
- LLM 검증 layer 실패 로그 (§5)
- admin 수동 reject 기사 (사용자가 invalidate 한 기사 → "왜?" 메모)

### 10-2. 분석 사이클 (분기 1회 / 사용자 → planner-architect 협업)
1. 👎 가 많은 기사 Top 20 추출
2. 신고 사유 분류 (점수 오류 / 선수명 오류 / 표현 어색 / 추측 / 기타)
3. 분류별 prompt 개선안 도출
   - 점수·이름 오류 → 검증 layer 강화
   - 표현 어색 → system prompt 룰 추가 (예: "이런 표현 피하기")
   - 추측 → few-shot examples 추가
4. prompt v1.0 → v1.1 업데이트 + 새 매치 100건 A/B 테스트
5. 통과 시 v1.1 배포 + DB `prompt_version` 추적

### 10-3. Few-shot 풀 확장
- 사용자가 직접 작성한 좋은 기사 → few-shot examples 추가 (현재 2건 → 분기마다 +2)
- 다양한 flow (comeback / close / blowout / OT / 트리플더블 등) 커버리지 확보

## 16. developer 주의사항 (Phase 1 진행 시)

1. **LLM 신규 도입**: 코드베이스에 Anthropic SDK 미사용. `@anthropic-ai/sdk` 신규 설치 필요. `claude-api` skill 활용 (prompt caching 필수).
2. **환경변수**: `ANTHROPIC_API_KEY` 추가 (`.env` + Vercel preview/prod 분리). NEXT_PUBLIC 절대 금지.
3. **API route**: `src/app/api/web/articles/generate/route.ts` 신규. `withAuth` 필요 (admin 또는 cron-secret). 일반 사용자 호출 차단.
4. **fallback 우선**: LLM 응답 1번이라도 검증 실패 → 옵션 A 템플릿 즉시 사용. 절대 사용자에게 에러 노출 안 함.
5. **DB schema 추가**: §6-2 `match_articles` 테이블 — Phase 3 에서 도입. `prisma db push` 운영 즉시 적용 (NULL 허용 ADD COLUMN 만 → 무중단).
6. **prompt 버전 관리**: `lib/articles/prompts/v1.0.ts` 형식으로 파일별 분리. git 으로 prompt 변경 추적.
7. **테스트 데이터**: 5/2 D-day 종료 후 실 매치 데이터로 단위 테스트 (`tests/articles/llm-reporter.test.ts`).
8. **rate limit**: `RATE_LIMITS.articleGenerate` 신규 (10/60s). 사용자 강제 재생성 spam 방어.
