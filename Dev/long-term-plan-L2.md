# L2. 본인 / 타인 프로필 시각 통합 (장기 과제)

> 작성: 2026-04-20 (W4 Day 20 회고 · `ux-implementation-plan-2026-04-19.md` L5 연장)
> 범위: 분기 단위 (~15시간 / 2주 분산)
> 진입 시점: 다음 분기 시작 시 — W5+ 또는 L3와 일정 조율

---

## 1. 배경

현재 프로필은 **"내가 보는 내 프로필"** 과 **"타인이 보는 내 프로필"** 이 서로 다른 페이지·다른 컴포넌트·다른 시각 구성으로 나뉘어 있다.

| 경로 | 대상 | 구조 | 현황 |
|------|------|------|------|
| `/profile` (허브) | 본인 | 좌측 네비 7항목 + 통합 대시보드(M1 Day 7+) | 최신. 편집 버튼, 맞춤 필터, 미완성 배너 포함 |
| `/users/[id]` | 타인(또는 본인) | 단일 페이지 카드 컴포넌트 | 디자인·정보 구조가 `/profile` 와 달라 일관성 깨짐 |

사용자 피드백: "내 프로필이 다른 사람에게는 어떻게 보이는지 한눈에 파악이 안 된다", "편집 후 결과물 확인이 번거롭다".

---

## 2. 설계 결정

### 결정 A — 단일 경로 통합 (추천)
`/users/[id]` 를 단일 진입점으로 두고, 본인이 본인 id 를 볼 때는 편집/비공개 정보를 함께 노출한다.
- 장점: 코드 중복 제거, 시각 통일 자연스러움, "자기 프로필을 타인 시점으로 보기" 버튼 UX 불필요
- 단점: `/profile` 대시보드의 맞춤 위젯(내 활동/다음 경기/주간 리포트 등)을 `/users/[id]` 에 어떻게 녹일지 정리 필요

### 결정 B — 경로 유지 + 공용 컴포넌트화
경로는 `/profile` (본인), `/users/[id]` (타인) 유지하되, **hero·stat·roster** 등 공통 섹션을 `src/components/profile-common/` 아래 공용 컴포넌트로 분리해 양쪽에서 import.
- 장점: 기존 경로/링크 호환 완벽
- 단점: "본인 시점 미리보기" UX 를 따로 구현해야 하고, 분기 로직이 컴포넌트마다 깔림

### 추천 = **A (단일 경로)** + `/profile` 은 "내 대시보드"로 재정의
- `/users/[id]` → **프로필 단일 소스**. 본인 접근 시 편집 버튼/비공개 정보 노출
- `/profile` → **대시보드 허브**. 프로필 요약(상단) + 맞춤 위젯(내 활동/다음 경기/주간 리포트/완성 배너). 좌측 네비 유지
- 내비게이션: 헤더 프로필 드롭다운 → "내 프로필 보기"는 `/users/[me.id]`, "대시보드"는 `/profile`

## 3. 영향 페이지 · 파일

### 직접 수정 (L2 본 작업)
- `src/app/(web)/users/[id]/page.tsx` — 본인/타인 분기 hero 도입, 편집 버튼 조건부, 공개 범위 처리
- `src/app/(web)/profile/page.tsx` — "내 프로필 요약 상단" + 기존 맞춤 위젯 유지. 네비 "내 정보" 라벨은 "대시보드"로 재검토
- `src/components/profile/profile-side-nav.tsx` — "내 프로필 보기" 외부 링크 추가 고려
- `src/components/profile/` 하위 위젯들 — hero·stat·roster 를 `users/[id]` 와 공유 가능한 형태로 리팩토링

### 연동 체크 (깨지면 안 됨)
- `/profile/edit`, `/profile/basketball`, `/profile/growth`, `/profile/activity` 등 서브 페이지 네비 호환
- 헤더 `ProfileDropdown` 링크 대상
- 팀원 목록에서 팀원 이름 클릭 시 이동 (`/users/[id]`) — 현재도 이 경로 사용
- OAuth/로그인 직후 리다이렉트(`/profile`) — 유지
- SEO: `/users/[id]` 메타 (공개 프로필 기준)

### 보안
- 공개 범위 필드 명시 (닉네임·팀 소속·공개 스탯 = 공개 / 전화번호·이메일·실명 = 비공개)
- 본인 접근 판별: `getWebSession()` 비교. 세션 없으면 비공개 필드 전부 가림

## 4. 예상 공수 (총 ~15h)

| 단계 | 작업 | 공수 |
|------|------|------|
| 1 | 현 `/users/[id]` 와 `/profile` 구조·공통 필드 차이 audit | 2h |
| 2 | `ProfileHero`·`ProfileStatBar`·`ProfileRoster` 공용 컴포넌트 설계/분리 | 3h |
| 3 | `/users/[id]` 에 본인 접근 분기 + 편집 버튼 + 비공개 필드 규칙 | 3h |
| 4 | `/profile` 을 "대시보드"로 재정의 (상단 요약 + 기존 위젯 유지) | 3h |
| 5 | 헤더 드롭다운/내비 라벨 조정 + 이동 경로 정합 | 1h |
| 6 | 다크/라이트·PC/모바일 시각 검증 + 기존 SEO/메타 회귀 | 2h |
| 7 | `knowledge/conventions.md` 에 "프로필 공개 범위" 규칙 기록 | 1h |

## 5. 선결 조건 / 리스크

- **선결**: M4(내 활동) 안정화 확인. `/profile/activity` 가 단일 소스라는 전제가 깨지면 대시보드 재정의가 무의미해짐 — 지표: 2주 이상 NullRef/401 0건
- **리스크**: 헤더 `ProfileDropdown` 링크와 팀 페이지 내 "이름 클릭 이동" 등 프로젝트 전반의 이동 경로를 한 번에 정합시켜야 해서 PR 크기가 커짐 → **PR 2개로 분할 권장** (a) 공용 컴포넌트 분리 + `/users/[id]` 리뉴얼, (b) `/profile` 대시보드 재정의
- **L3(IA 명확화) 와 교차**: L3 에서 단체/시리즈/대회 브레드크럼을 도입할 때 "사용자의 소속 팀·단체 표기" 가 프로필에 노출될 수 있음 → L3 먼저 확정 후 L2 착수 순서 권장
