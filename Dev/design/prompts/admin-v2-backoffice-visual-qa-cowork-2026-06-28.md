# 코워크 의뢰 — admin-v2 백오피스 시각충실도 QA (시안 픽셀 대조)

> **목적**: 새로 그린 관리자 `/v2` 백오피스가 시안 정본 대비 "깔끔함·완성도"가 부족하다는 느낌의 **구체적 원인을 요소 단위로 식별**한다. CLI(코드 박제)가 바로 고칠 수 있는 형태로 차이 목록을 산출.
> **역할 분담**: 코워크=시각 QA(이 의뢰) / CLI=차이 수정 / Claude.ai=시안 원본.

---

## 1. 배경 (꼭 읽기)

- 관리자 영역을 **클린 슬레이트로 재구축** 중. 전제 = **시안 HTML을 verbatim/1:1로 충실 포팅**(레거시 0의존).
- 진행: R1 토대(정본 `toss.css`+`admin-pages.css` 값 그대로 + `toss-kit`/`admin-shell`/`admin-blocks` 1:1 포팅) → **쇼케이스 `/v2/_showcase` 사용자 합격**. 이어 R2-A 백오피스(셸/IA 5콘솔 + 유저 콘솔) 구현.
- 사용자 피드백: "동작은 OK, **거의 비슷한데 시안보다 깔끔함이 떨어지고 완성도가 부족**."
- **이미 적용된 수정(H1)**: 전역 `globals.css`의 `body` 타이포(font-size 14~15px·line-height 1.55·font-feature-settings tnum)가 상속으로 `/v2`에 새어들던 누수를, `[data-admin="v2"]` 루트에 `font-size:16px; line-height:normal; font-feature-settings:normal`로 차단. → **이 수정으로 텍스트가 촘촘해졌는지 우선 확인**하고, 남은 차이를 찾아라.

## 2. 대조 대상

## 3. 원칙

**정본 1:1 충실 포팅이 전제 → 모든 이탈 = 버그.** 사소해 보여도(간격 2px·그림자 농도·radius·폰트 크기/굵기·active 색·아이콘 정렬·뱃지 톤) **누적되면 "덜 깔끔"**의 정체다. 추측 말고 시안 스크린샷·토큰값과 직접 대조.

## 4. 점검 체크리스트 (요소별·구체적)

- **타이포**: 본문/제목/숫자 폰트 크기·줄간격·굵기·자간. (H1 수정 후 시안만큼 촘촘한가? 숫자 비례형인가?)
- **사이드바**: 폭(정본 248px)·그룹 라벨(운영/회원·팀/대회·경기) 스타일·active 항목 배경/색·아이콘·간격
- **KPI 카드**: 그림자(`--sh-*`)·radius(card 24)·패딩·아이콘 타일(둥근 사각+틴트bg)·**delta pill**(+8%/−11% 컬러)·숫자 크기/굵기
- **패널/차트/리스트 카드**: 헤더·구분선·여백·막대그래프 색/모양·리스트 행 간격
- **색상**: Toss Blue `#3182F6`·회색조(--ink/--ink-soft/--ink-mute)·배경(--bg #F2F4F6)·테두리(--border #E5E8EB)
- **컴포넌트**(유저 콘솔): 탭·테이블 행/헤더·뱃지·버튼·토글·모달이 시안 토큰과 동일한가
- **상태**: hover·active·텍스트 선택색·포커스
- **전반 리듬**: 카드 간 간격·섹션 여백·정렬이 시안과 같은 호흡인가

## 5. 산출물 (CLI 전달용 — 이 형식으로)

요소별 차이를 **우선순위(High/Med/Low)** 로:

| 우선순위 | 화면/요소 | 시안 모습(값) | 현재 모습(값) | 추정 원인 / 수정 방향 |
|---|---|---|---|---|
| 예) High | 대시보드 KPI 카드 | 그림자 `--sh-sm` 또렷·radius 24 | 그림자 거의 없음·radius 16 | `.ad-kpi` box-shadow/radius 토큰 누락 의심 |

- 가능하면 **렌더 vs 시안 비교 스크린샷**(나란히·주석) 첨부.
- 마지막에 **"가장 임팩트 큰 차이 Top 3"** 요약.
- H1(타이포) 수정이 충분했는지 명시(촘촘해졌나 / 여전히 부족하면 어디).

## 6. 주의

- 데이터 차이(예: KPI 우측 2개 delta pill 없음 = 상태 스냅샷이라 의도적 생략, "준비 중" 플레이스홀더 = 백엔드 데이터 없음)는 **시각 결함 아님** — 구분해서 보고.
- 관리자=Toss 디자인시스템(lucide·Toss Blue). 공개 사이트(BDR 13룰)와 무관.
- **코드/DB는 건드리지 말 것** — 시각 차이 식별·보고만. 수정은 CLI가 한다.
