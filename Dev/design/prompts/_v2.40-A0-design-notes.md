## 1. 🔴 정산 집계 쿼리 설계 (A4 `tournament:id` 정산 탭 대비)

### 1-0. 시안 정산은 mock — 실측과 구조가 다름

시안(`au-detail.jsx` L253~268) 정산 탭 = `TN_SETTLE.rows` 더미 + DL 하드코딩(`수입 4,480,000 / 지출 2,340,000 / 순이익`). **숫자 전부 mock**. 아래는 실측 schema 기반 산출 설계.

### 1-1. 대회 수입 (Revenue) — `payments` 아님, `TournamentTeam` 기반

🚨 **핵심 함정**: `payments.payable_type`의 실제 사용값은 **`"Plan"`(구독결제) / `"CourtBooking"`(코트예약)** 둘뿐(src 전수 grep 확정). **대회 참가비 결제 레코드는 `payments`에 존재하지 않음**. 대회 참가비는 운영자가 **계좌 입금 확인 후 `TournamentTeam.payment_status`를 수동 전환**하는 오프라인 입금 모델(schema L340~342 `bank_name/bank_account/bank_holder` = 입금 계좌 안내).

| 항목 | 출처 모델·필드 | where / 조건 | 산출식 |
|------|--------------|-------------|--------|
| **승인 참가팀** | `TournamentTeam` | `tournamentId = :id` AND `status` 승인계열(`approved`/`active` 등 — §주의1) | count |
| **입금 완료 팀** | `TournamentTeam.payment_status` | 위 + `payment_status = 'paid'` (기본값 `'unpaid'`) | count |
| **팀별 참가비** | `Tournament.entry_fee`(Decimal, default 0) **또는** `Tournament.div_fees`(Json `{종별:금액}`) | 팀의 `division`/`category`로 div_fees 우선 룩업, 없으면 entry_fee 폴백 | per-team fee |
| **수입 합계** | 위 조합 | `payment_status='paid'` 팀들의 fee 합 | Σ(paid팀 fee) |

- **종별 차등 참가비**: `Tournament.div_fees`(Json, schema L336) = `{디비전명: 금액}`. 팀의 `TournamentTeam.division` 또는 `category`(L576/596)로 룩업. div_fees 비어있으면 단일 `entry_fee`(L300) 적용.
- ⚠️ **주의1 (status 값 실측 필요)**: `TournamentTeam.status` default `'pending'`. 승인 상태값(`approved`/`active`/`confirmed` 등)은 A4 착수 시 **실데이터 distinct 1회 확인** 후 where 확정(억측 금지). 입금 산출은 `payment_status='paid'` 단일 기준이라 status와 독립.
- ⚠️ **주의2 (Decimal 직렬화)**: `entry_fee`·div_fees 금액은 Decimal/숫자 → `Number()` 변환 필수(BigInt 아님). `TournamentTeam.id`/`teamId`는 BigInt → `.toString()`.
- **payments 보조 확인(범위 밖)**: 만약 향후 일부 대회가 PG(토스) 참가비 결제를 도입하면 `payments WHERE payable_type='Tournament'`가 생길 수 있으나 **현재 0건**. A4는 TournamentTeam 기반으로만 산출하고, payments 경로는 "현재 미사용" 주석.

### 1-2. 대회 지출 (Expense) — 전용 모델 부재, 심판비만 실측 가능

🔴 **지출 전용 모델 부재 확정**: schema 전수 grep — `Cost`/`Expense`/`Spend`/`Disburse`/`Outlay`/`Payout` 모델 **0건**. 대회 운영 일반 지출(대관료·물품·상금 등)을 담는 테이블 없음.

| 지출 항목 | 출처 모델 | 대회 연결 경로 | 산출 가능? |
|----------|----------|--------------|:---------:|
| **심판 정산비** | `RefereeSettlement.amount`(Int, 원) | `RefereeSettlement.assignment_id` → `RefereeAssignment.tournament_match_id` → `TournamentMatch.tournamentId = :id` | ✅ **가능** |
| 대관료 / 물품 / 상금 / 기타 운영비 | — (모델 없음) | — | 🔴 **보류 (수동입력)** |

- **심판비 지출 산출식**: `RefereeAssignment WHERE tournament_match_id IN (해당 대회의 TournamentMatch.id 들)` → 그 assignment들의 `RefereeSettlement.amount` 합. (`RefereeAssignment.fee`(L2873, nullable)도 개별 배정비이나, 확정 정산은 `RefereeSettlement` 1:1 — `@@unique(assignment_id)`). 상태 필터: `RefereeSettlement.status`(`pending`/`paid` 등)로 "지급예정 vs 지급완료" 구분 표시 가능.
- ⚠️ `RefereeAssignment`↔`TournamentMatch`는 **relation 미선언**(schema L2877 주석 "tournament_match 쪽 관계 선언 없음 — TournamentMatch 미수정"). → 조인 불가, **2단계 쿼리**: ①대회의 `TournamentMatch.id[]` 조회 → ②`RefereeAssignment WHERE tournament_match_id IN (...)` → ③assignment_id로 `RefereeSettlement` 합산. (또는 `prisma.$queryRaw`).

### 1-3. 순이익 + 산출 가능 범위 / 보류 범위 명확 구분

| 구분 | 산출 | 범위 |
|------|------|------|
| ✅ **수입** | Σ(입금완료 팀 × 종별/기본 참가비) | **산출 가능** (TournamentTeam.payment_status + entry_fee/div_fees) |
| ✅ **지출(심판비)** | Σ RefereeSettlement.amount (대회 경기 배정분) | **산출 가능** (2단계 쿼리) |
| 🔴 **지출(운영비)** | 대관·물품·상금·기타 | **DB 부재 → "수동 입력 보류" / "—" 표기** |
| ⚠️ **순이익** | 수입 − 심판비 (− 수동입력 운영비) | **부분 산출** — "심판비만 반영, 기타 운영비 미집계" 명시 필수 |

📌 **A4 정산 탭 UI 권고**:
- 수입 카드 = 실측(입금완료 N팀 / 미입금 M팀 / 수입 합계 ₩X) — **신뢰**.
- 지출 카드 = 심판비 실측 + "기타 운영비(대관·물품 등)는 미집계 — 수동 입력 예정" 안내 배지.
- 순이익 = `수입 − 심판비` 표기하되 **"운영비 미반영 추정치"** 라벨. 시안의 "정산 확정" 버튼(L266)은 **D2(읽기요약)에 따라 미구현 or `tournament-admin` 위임**(콘솔은 읽기 전용).
- 보류분은 시안처럼 숫자 박지 말고 **"—"** 표기(errors.md "시안 정산 수치 mock" 룰).

## 2. 🟡 키트 컴포넌트 API 확정 (A2 대비)

### 2-1. ★ 핵심 — 기존 `admin-toss/` 키트와 중복 (신규 최소화)

기존 `src/components/admin-toss/`(`kit.tsx` + `data.tsx`, barrel `index.ts`)에 v2.40 9 프리미티브 중 **6개가 이미 존재**. A2는 "9개 신규 박제"가 아니라 **기존 재사용 + 부족분 4개 추가**가 정답.

### 2-2. Next.js 서버/클라 컴포넌트 경계 (시안 jsx → src 재구현)

시안 au-kit.jsx는 전역 스코프 React+Babel(`window.X`). src 박제 시 'use client' 분리:

| 프리미티브 | 경계 | 사유 |
|-----------|------|------|
| `PageHead` | **서버 가능** | 순수 표시(props만). 단 actions에 client 버튼 children 주입 가능 |
| `StatRow` | **서버 가능** | StatCard 래퍼·순수 표시 |
| `Panel` | **서버 가능** | 카드 래퍼·순수 |
| `DL` | **서버 가능** | key/value 리스트·순수 |
| `PrimaryCell` | **서버 가능** | 아바타+텍스트·순수 |
| `StatusBadge`/`Badge` | **서버 가능** | 순수 (기존 Badge가 이미 그러함) |
| `DataTable` | **`'use client'`** | onRowClick·정렬·페이지네이션 상호작용 (기존도 client) |
| `Toolbar` | **`'use client'`** | onSearch·onTab 핸들러 |
| `Drawer` | **`'use client'`** | open/onClose state·ESC keydown effect |
| `useFilter` 훅 | **`'use client'`** | useState/useMemo |

- **패턴**: 서버 페이지가 prisma로 `rows` 조회 → 서버 컴포넌트(PageHead/StatRow) 직접 렌더 + client wrapper(`<XxxClient rows={...} />`)에 직렬화 데이터 주입 → client wrapper 안에서 useFilter+Toolbar+DataTable+Drawer. **기존 admin Toss 화면들의 server/client 분리 패턴 답습**(예: `*-client.tsx`).
- **직렬화 주의**: 서버→client props는 BigInt `.toString()` / Date ISO / Decimal `Number()`(snake_case 주의 — server 직접 prisma는 camel, API 경유는 snake. errors.md 7회 함정).

### 2-3. DataTable columns 스펙 (확정 — 기존 admin-toss와 통일)

기존 `Column<T>`(data.tsx L196) = `{ key, label, width?, align?:'left'|'center'|'right', sortable?, render?(row) }`. 시안 au-kit DataTable columns = `{ key, label, w, align:'l'|'c'|'r', cls, hideSm, render(row) }`.

**확정 = 기존 `Column<T>` 채택 + `hideSm` 추가**:
```ts
type Column<T> = {
  key: string;
  label: React.ReactNode;
  width?: number | string;   // 시안 w → width
  align?: "left"|"center"|"right";  // 시안 l/c/r → left/center/right
  sortable?: boolean;        // 기존 강점 유지
  hideSm?: boolean;          // ★ 신규 추가 (시안 hideSm — 모바일 숨김)
  render?: (row: T) => React.ReactNode;
};
```
→ A2에서 기존 `Column<T>`에 `hideSm?: boolean` 1필드만 추가하면 시안 DataTable 요구 100% 충족. **신규 DataTable 만들지 말 것**.

### 2-4. Drawer vs 별도 라우트 상세 경계 (A3 ↔ A4)

- **A3(목록)**: 행 클릭 → **Drawer**(우측 요약 + "상세 페이지 열기" 링크). Drawer는 요약만(DL 2~3행 + 핵심 액션).
- **A4(드릴다운 상세)**: 풀 상세는 **별도 라우트** `/admin/<base>/[id]`(탭 구성). Drawer의 "상세 열기"가 여기로 link.
- → Drawer = 가벼운 미리보기, 라우트 상세 = 풀 탭. **둘 다 필요**(시안도 Drawer→풀상세 2단). A2는 Drawer만, A4는 라우트 상세 신설.

## 3. 🟢 19섹션 ↔ 데이터 출처 매핑표

**판정 핵심**: 10화면 = **Server Component 직접 prisma**(camelCase·force-dynamic·직렬화 후 client wrapper에 props) / 8화면 = **Client + API fetch**(snake_case·useEffect+fetch). **요약 stat은 절반 이미 존재**(groupBy 등) — A3 StatRow는 가능한 한 기존 쿼리 재사용, 신규는 count/groupBy 소량 추가만.

| # | 라우트 | 데이터 출처 | 주요 모델·쿼리 | StatRow 요약 stat | 행 액션 |
