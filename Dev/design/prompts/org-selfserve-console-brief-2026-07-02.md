## ★ 이 의뢰의 한 줄 정의

> **협회/연맹 운영자(단체 owner·admin)가 자기 계정으로 직접 로그인해 "자기 단체만" 스스로 관리하는 self-serve 운영 콘솔**을 설계한다.
> 파트너사가 `/partner-admin`에서 자기 광고·대관을 스스로 관리하듯, 단체 운영자도 전용 콘솔에서 멤버·팀·회원·하위단체·대회·회비·공지를 스스로 처리한다.
> **super_admin이 모든 단체를 승인/관리하는 `/admin/organizations`(승인 큐)와는 완전히 별개** — 이건 "단체 본인의 콘솔"이다.

---

## ★ 설계 원칙 — 점진적 온보딩 (Progressive Onboarding) · 전반 적용

> 06-29 브리프의 제품 철학을 그대로 계승. **모든 화면이 이 톤을 1순위로 지킬 것.**

1. **초기 진입은 무조건 간편하게** — 최소 정보로 단체 즉시 생성. 첫 화면 "장벽 0".
2. **서류·인증·계좌는 "나중에"** — 실제로 필요해지는 시점(정산·공식대회)에만 부드럽게 유도. 인증 미뤄도 **기본 사용은 막지 않음**(강요 X, 유도 O).
3. **잠금은 "실질 결제/법적" 기능에만** — 단체 운영·멤버·시리즈·공지 등은 인증 없이 그대로. 잠기는 건 정산 출금·공식대회 자격 정도.

```
[즉시 생성/진입] 최소 필드로 단체 운영 시작 — 서류/인증/계좌 요구 0
      │
      ▼ (쓰다가 아래 시점 도달 시)
[deferred 인증 유도]
      ├─ 참가비·회비 정산 / 출금 계좌 필요할 때
      ├─ 공식 대회 개최(법적 효력) 필요할 때
      └─ 그 외 증빙 필요 시점
```

## 0. 디자인 시스템 = **admin-v2** (혼동 금지)

이 콘솔은 **self-serve 운영자 콘솔**이므로 **admin-v2 계열**을 사용한다. 공개 단체사이트(BDR v2·13룰)와 **완전히 다른 시스템**이다.

| 항목 | 이 콘솔(self-serve) | 공개 단체사이트(06-29 Part1) |
|------|--------------------|------------------------------|
| 디자인 시스템 | **admin-v2** (`components/admin-v2` + `styles/admin-v2`) | BDR v2 |
| 아이콘 | **lucide** (kit `ICON_ALIAS`, kebab→Pascal) | Material Symbols Outlined |
| 셸 | AdminShell 계열(사이드 레일 + 패널) | AppNav frozen |
| 13룰 | **미적용** | 전부 적용 |
| 라운딩·색 | admin-v2 토큰 따름(웹 4px 룰 아님) | var(--*) BDR 토큰 |

> ⚠️ 정본 참조: 기존 admin-v2 화면(`/v2`, `/v2/ta/*`, `/v2/operate/*`)과 **시각적으로 동일 계열**이어야 한다. `PageHead`·`DataTable`·`Btn`·`Icon`·`Modal`·`Empty`·`ad-card*` 정본 컴포넌트를 그대로 재사용. Material Symbols / radius 4px / AppNav를 여기 적용하면 정본 어긋남.

## 1. 컨텍스트

- **운영 주체**: 단체 owner / admin (self-serve). member 롤은 콘솔 진입 불가(열람 대상일 뿐).
- **관리 대상 4종**: ①멤버·임원(직책) ②소속 팀 ③선수/개인 회원(가입신청 승인) ④산하 하위단체(상위→지부 계층)
- **고유 기능 3종**: ①대회·리그 개최 ②회비·정산 ③공지·소통
- **우선순위**: P1 (사용자 요청)
- **시급도**: 이번 batch = **시안만**. 박제·DB·결제는 전부 후속(별도 승인).

### 1-1. 현황 한 줄

- 단체 도메인은 **기본 백엔드(생성/수정/멤버/시리즈/공개사이트)가 완비**돼 있으나, self-serve 운영자용 콘솔은 **읽기전용 stub**(`/v2/ta/organizations` — 모든 버튼 "준비 중" 토스트)뿐이다.
- 이번 요청의 상당 부분(소속팀·가입신청·하위단체·직책·회비정산·공지)은 **현재 DB에 없는 신규 개념**이다 → §2 백엔드 경계 3단계 분류 참조. **시안은 만들되, 신규 백엔드는 후속 설계 + 별도 사용자 승인.**

### 1-2. ✅ 사용자 결정 확정 (2026-07-02 수빈 확정 — 디자인 클로드는 이대로 진행)

| # | 항목 | 확정값 | 반영 위치 |
|---|------|--------|----------|
| 1 | **진입 라우트** | **`/org-console`** (파트너 콘솔 미러·글로벌 단일진입→내 org 자동선택·다중시 스위처). `/v2/org/[slug]`·stub 확장은 기각 | §3-2 |
| 2 | **P1 첫 착수 화면** | **대시보드 + 단체 정보(로고·배너 브랜딩 편집 포함)**. 멤버·임원/대회·리그는 P1이나 후속 배치 | §3-1·§4-1 |
| 3 | **하위단체 계층 깊이** | **1단(상위→직속 지부)만**. 다단계 재귀 트리 UI 배제(후속 여지만) | §4-5 |

## 2. 데이터 기준 — 실제 DB vs 신규 컨셉 (백엔드 경계 3단계·필수)

> ⚠️ 각 기능을 **[기존 DB로 가능] / [무중단 ADD로 가능] / [신규 테이블 필요·큰 작업]** 3단계로 분류했다. 디자인 클로드는 이 표를 보고 **어떤 화면이 "실동작 목표" vs "컨셉(미구현) 제안"인지** 인지하고 그릴 것.

### 2-1. ✅ 실제 DB 컬럼 (지금 있는 것)

**`organizations`**: name / slug(수정불가·표시만) / logo_url / banner_url / description / region / contact_email / contact_phone / website_url / owner_id / status(pending·approved·rejected·archived) / is_public / series_count / apply_note / 승인메타(approved_at·approved_by·rejection_reason·rejection_at) / created_at
**`organization_members`**: role(**owner·admin·member 3종만**) / is_active / user(nickname·email·profile_image_url) / @@unique[org,user]
**`tournament_series`**: name / slug / description / logo_url / status / tournaments_count / **organization_id?(nullable FK)**
**`Tournament`**: edition_number / venue_name / status / champion_team_id / startDate

### 2-2. 백엔드 경계 3단계 분류표 (핵심)

| # | 기능(화면) | 관리대상/고유 | 경계 분류 | 근거·방법 |
|---|-----------|--------------|----------|----------|
| 1 | 단체정보·브랜딩 편집(로고/배너/연락처/공개) | — | **[기존 DB]** | PATCH `/api/web/organizations/[id]` 완비(로고/배너 URL 저장). 단 **파일 업로드 위젯**은 스토리지 인프라 후속(06-29 §4-4) |
| 2 | 멤버 목록·초대·제거 | ①멤버 | **[기존 DB]** | members GET/POST/DELETE 완비. **단 role 변경 PATCH는 미지원** |
| 3 | 임원 직책 세분화(회장/부회장/총무) | ①임원 | **[무중단 ADD]** | `organization_members.position VARCHAR?` 추가(무중단) 또는 role 확장. 현재는 owner/admin/member뿐 |
| 4 | 소속 팀 관리(직접 등록/연결) | ②팀 | **[신규 테이블·큰작업]** | **org↔team 직접관계 부재**. 시리즈→대회→팀만 존재 |
| 4b | 소속 팀 "읽기"(참가팀 역추적 집계) | ②팀 | **[기존 DB·부분]** | 시리즈→대회→TournamentTeam 역추적 집계는 읽기 가능(신규 집계 로직 필요·쓰기 불가) |
| 5 | 회원 가입신청 승인/반려 | ③회원 | **[신규 테이블·큰작업]** | **유저→단체 가입신청 모델 부재**. `team_join_requests`(팀용)는 유사 참조 패턴 |
| 6 | 산하 하위단체 계층 | ④하위단체 | **[무중단 ADD]** | `organizations.parent_org_id BigInt?`(self-FK) 무중단 추가. 깊이는 결정 필요(1~2단 권장) |
| 7 | 대회·리그 개최 | 고유① | **[기존 DB]** | tournament_series/Tournament + **마법사(`/v2/ta/tournaments/new`) 완비**. 콘솔은 진입 링크·목록만 |
| 8 | 회비·정산(참가비/회비 징수·출금) | 고유② | **[신규 테이블·큰작업 + 결제]** | **org 정산 모델 부재**. `PartnerSettlement`(파트너용·period/gross/fee/net/status/scheduled_at/paid_at) = 참조 스키마. 출금계좌·PG 연동 후속 |
| 9 | 공지·소통(공지·멤버 알림·메시지) | 고유③ | **[신규 테이블·큰작업]** | **org 공지/메시지 모델 부재**. notifications는 개인용. org-scoped 공지 신규 |
| 10 | 인증·서류 상태(배지/체크리스트/잠금) | — | **[신규 테이블·큰작업]** | 06-29 §4-5 O7. 인증상태·서류제출·계좌 모델 후속 |
| 11 | self-serve 권한 게이트 | — | **[기존 DB]** | organization_members role owner/admin + **partner-admin/layout.tsx 서버 게이트 패턴** 재사용 |

### 2-3. ❌ 절대 그리지 말 것 (06-29 유지)

설립연도(founded_year)·본부주소(address)·운영원칙·스폰서·브랜드색상(brand_color) 컬럼 없음 → mock 금지.
색상은 `org-color.ts` pickColor(id) 해시 팔레트(6색)로만.

> ✅ **의도적 예외 = "컨셉(미구현)" 성격으로 시안화 허용**: 위 표의 **[무중단 ADD]·[신규 테이블] 분류 기능**(직책·소속팀·가입신청·하위단체·회비정산·공지·인증서류)은 DB에 없지만 **"목표 UX 컨셉"으로 일부러 그린다**. 각 화면 상단에 **"컨셉(준비 중)" 라벨/톤**을 반드시 부여하고, 샘플 상태는 컨셉 데이터로 표현("실데이터 와이어 가정 금지"). 반면 §2-3 절대금지 필드는 컨셉이라도 그리지 말 것(둘을 혼동 금지).

## 3. IA · 진입 라우트 · 권한 게이트

### 3-1. 콘솔 네비게이션(좌측 레일) — admin-v2 셸

| 순위 | 섹션 | 경계 | 내용 |
|------|------|------|------|
| **P1 첫 착수** | **대시보드** | 기존 DB | 요약 KPI(멤버수·시리즈수·진행중 대회·인증 진행률) + 최근 활동 + 인증 유도 배너 |
| **P1 첫 착수** | **단체 정보** | 기존 DB | 기본정보 조회/편집 + **로고·배너 편집(브랜딩)** + 인증·서류 현황(컨셉) |
| P1 | **멤버·임원** | 기존 DB + 직책=무중단ADD(컨셉) | 멤버 목록/초대/제거(실동작) + **직책 텍스트 부여**(컨셉·position) |
| P1 | **대회·리그** | 기존 DB | 시리즈 목록 + 회차 통합표 + **"대회 개최" = 마법사 진입 링크** |
| P2 | **하위단체** | 무중단ADD(컨셉) | 상위→직속 지부 리스트(1단) + 하위단체 등록(컨셉) |
| P2 | **공지·소통** | 신규(컨셉) | 공지 목록/작성 + 멤버 알림 발송(컨셉) |
| 후속 | **소속 팀** | 신규(컨셉) + 읽기부분 | 소속/참가 팀 목록(컨셉) |
| 후속 | **회원 신청** | 신규(컨셉) | 가입신청 승인/반려 큐(컨셉) |
| 후속 | **회비·정산** | 신규+결제(컨셉) | 징수/정산/출금 + 잠금 게이트(컨셉 시안 only) |
| — | **설정** | 기존 DB | is_public 토글 + 권한 위임(UI만) + 단체 보관/삭제 |

> ✅ **P1 첫 착수 화면(확정 2026-07-02) = 대시보드 + 단체 정보(로고·배너 브랜딩 편집 포함)**. 이 2개를 먼저 완성해 콘솔 골격 확정. 멤버·임원/대회·리그는 P1(기존 DB)이나 후속 배치. 무중단 ADD(하위단체·직책·공지)=P2 컨셉. 신규 테이블+결제(소속팀·가입신청·회비정산)=후속 — **이번엔 레일 자리·빈 골격/잠금 컨셉만**.

### 3-2. 진입 라우트 = **`/org-console`** (✅ 확정 2026-07-02)

- **`/org-console`** — 파트너 콘솔(`/partner-admin`) 미러. 글로벌 단일 진입 → 내 소속 org 자동 선택 → **다중 소속 시 org 스위처**. self-serve "내 단체 콘솔" 정체성이 명확하고, 컷오버 진행 중인 `/v2/ta` 셸과 얽히지 않음.
- 대안(`/v2/org/[slug]`·기존 `/v2/ta/organizations` stub 확장)은 **기각** — `/v2/ta`는 super/tournament_admin 스코프라 self-serve 정체성이 흐려짐. **결정: `/org-console` 채택.**

### 3-3. 권한 게이트 = **owner/admin만 진입** (✅ 확정 2026-07-02·partner-admin 패턴 미러)

```
getWebSession() → null? redirect("/login")
organization_members.findFirst({ user_id, is_active, role in [owner, admin] })
  → 없음 & !super_admin ? redirect("/")   // member·비소속 진입 차단
super_admin = 전능 우회(첫 active org 자동선택 or sentinel)
→ org 컨텍스트 확정(단일 자동선택 / 다중 스위처)
```

- **owner/admin만 콘솔 운영** — **member는 콘솔 진입 불가**(공개 단체사이트 `/organizations/[slug]`로만 열람). 서버 컴포넌트 레이아웃에서 강제(클라 신뢰 X).
- 상단 헤더에 단체명 + **인증 상태 배지**(컨셉) + 승인상태(pending 시 warn 뱃지).

## 4. 화면별 상세 명세

> 각 화면은 admin-v2 정본(PageHead·DataTable·Btn·Icon·Modal·Empty) 재사용. 컨셉 기능은 상단 "준비 중" 톤 라벨 필수.

### 4-1. 대시보드 [기존 DB]

- 상단 헤더: 단체명 + 로고 + 인증 상태 배지(컨셉) + 공개/비공개 뱃지.
- KPI 카드 4종: 멤버 수(실) / 연결 시리즈 수(series_count 실) / 진행중 대회 수(실) / **인증 진행률(컨셉·N/M)**.
- 점진적 온보딩 배너: "서류·인증은 필요할 때 안내드려요"(안심 톤·닫기 가능).
- 최근 활동 리스트: 활동 로그 테이블 부재 → 정본 Empty "준비 중" 또는 최근 멤버 가입/대회 개설 근사.

### 4-2. 관리대상① 멤버·임원 [멤버=기존 DB / 직책=무중단ADD 컨셉]

- 멤버 목록 DataTable: 프로필 이미지·닉네임·이메일·**롤 뱃지(주최자/운영진/멤버)**·가입일·is_active.
- 초대 모달(실동작): 이메일 입력 + role 선택(owner 초대 불가) → POST.
- 제거(confirm) → 소프트 삭제(is_active=false).
- **직책 세분화(컨셉·✅ 확정: `position` 자유텍스트 필드)**: 회장/부회장/총무/이사 등 직책 뱃지 + "직책 입력/부여" 액션. **role enum 확장이 아니라 `organization_members.position VARCHAR?`(무중단 ADD) 자유텍스트 입력** 방식 — 텍스트 입력 필드(드롭다운 아님)로 그릴 것. 현재 role 3종(주최자/운영진/멤버) 뱃지 위에 직책 텍스트를 얹는 개념 시각화. 상단에 "직책 관리는 준비 중" 톤.
- role 변경 토글은 **표시만**(백엔드 미지원).

### 4-3. 관리대상② 소속 팀 [신규 테이블 컨셉 / 읽기 부분]

- **컨셉(준비 중) 화면**. org↔team 직접관계 부재 명시.
- 2블록 제안: ⓐ "소속 팀"(직접 등록·컨셉·신규 테이블) ⓑ "우리 대회 참가 팀"(시리즈→대회 역추적 읽기·부분 가능).
- 팀 카드: 로고·팀명·지역·소속 상태 뱃지. 등록 CTA = disabled + "준비 중" 안내.

### 4-4. 관리대상③ 회원 가입신청 승인 [신규 테이블 컨셉]

- **컨셉(준비 중) 화면**. 유저→단체 가입신청 모델 부재.
- 신청 큐 DataTable: 신청자 프로필·신청일·메모·상태(대기/승인/반려) + 승인/반려 액션(컨셉·disabled).
- 참조 패턴: `team_join_requests`(팀 가입신청) UX 톤 차용.
- Empty 상태 + "가입 신청 기능은 준비 중" 안내.

### 4-5. 관리대상④ 산하 하위단체 [무중단ADD 컨셉·✅ 확정: 1단(상위→지부)]

- **컨셉(준비 중)**. `parent_org_id` self-FK 하나 무중단 추가 예정.
- **깊이 = 1단으로 확정**: 상위 협회 → 직속 지부/지역협회 목록(단순 리스트/카드 그리드). **다단계 재귀 트리 UI는 배제**(그리지 말 것). 각 하위단체 = 단체명·지역·멤버수 요약 카드.
- "하위단체 추가" CTA(컨셉·disabled). "1단계(직속 지부)만 지원" 안내.
- (후속 여지: 다단계 계층은 필요 시 별도 설계 — 이번 시안 범위 밖.)

### 4-6. 고유기능① 대회·리그 개최 [기존 DB]

- 시리즈 목록(tournament_series·organization_id 연결) + 회차(Tournament) 통합표(edition_number·status·startDate·venue_name·champion).
- **"대회 개최" 버튼 = 기존 마법사 진입 링크**(`/v2/ta/tournaments/new`) — 콘솔은 개최 트리거·목록·상태 조회만, 실제 생성은 마법사가 담당. 새 폼 재설계 금지.
- 시리즈 추가 + "기존 대회 가져오기/이동" 액션(레거시 Absorb/Move 흐름 시각화).

### 4-7. 고유기능② 회비·정산 [신규 테이블 + 결제 컨셉·✅ 확정: 컨셉 시안 only]

- **컨셉 시안 only(준비 중·잠금 게이트)**. org 정산 모델 부재. **실제 PG 결제·출금계좌·DB는 전부 후속 별도 승인** — 이번엔 목표 UX 화면만 그린다(와이어링·실데이터 가정 금지).
- 정산 목록(PartnerSettlement 스키마 톤 참조): 기간(연·월)·건수·총액·수수료·정산액·상태(대기/입금완료)·입금예정일.
- **잠금 배너(점진적 온보딩 핵심)**: "참가비·회비 정산을 받으려면 출금 계좌·사업자 정보가 필요해요 → 지금 등록하기" + "나중에". 자물쇠 아이콘.
- 회비 징수 설정(참가비·연회비) 컨셉 카드. **결제/PG 연동은 후속(별도 사용자 승인·최대 작업).**

### 4-8. 고유기능③ 공지·소통 [신규 테이블 컨셉]

- **컨셉(준비 중)**. org 공지/메시지 모델 부재.
- 공지 목록 + 작성 모달(제목·본문·대상 멤버 범위). 발행/초안 상태.
- 멤버 알림 발송(컨셉·notifications는 개인용이라 org-scoped 신규 필요).
- 참조: 게시판(community_posts) 카피 톤.

### 4-9. 점진적 온보딩 — 인증·서류 deferred UX (★ 하이라이트·06-29 §4-5 계승)

3종 패턴을 **컨셉(미구현)**으로 시안화:

1. **단체 인증 상태 배지**: `기본 단체(미인증)` → `서류 제출중` → `인증 완료`(+반려). 대시보드/헤더 노출. 미인증도 **중립 톤**(경고 아님·"인증은 보너스").
2. **잠긴 기능 안내 배너**: 정산 출금·공식대회 개최에만 자물쇠 + "지금 등록"/"나중에" 2지선다. **기본 운영은 안 막음**.
3. **인증·서류 현황 체크리스트**: 대표자 인증 / 단체 증빙 / 출금 계좌 / 공식대회 자격. 각 미완료·검토중·완료 + "제출하기" CTA + 진행률(N/M). "지금 다 안 해도 괜찮아요" 톤.

## 5. States (전 화면 필수)

- **empty**: 데이터 0건 → 정본 Empty("등록된 X가 없습니다" + 안내). 더미 채우기 금지.
- **loading**: 정본 스켈레톤/스피너.
- **success**: 저장 토스트.
- **error**: 에러 토스트/인라인.
- **disabled**: 컨셉·미지원 버튼 = disabled + "준비 중" 툴팁/토스트.
- **잠금(locked)**: 인증 필요 기능 = 자물쇠 오버레이 + 안내 배너(§4-9 ②).
- **pending(승인대기)**: 단체 status=pending 시 헤더 warn 뱃지.

## 6. 백엔드 경계 · 후속 승인 필요 목록 (컷오버·박제 전 필독)

> 아래는 **시안만 제작·박제/DB/결제는 전부 후속 + 별도 사용자 승인**. §2-2 표의 [무중단 ADD]·[신규 테이블]·[결제] 항목 전부 해당.

| 후속 항목 | 경계 | 승인/작업 필요 |
|-----------|------|---------------|
| 직책 세분화 | 무중단 ADD | `organization_members.position` 추가 or role 확장 — 스키마 결정 |
| 산하 하위단체 | 무중단 ADD | `organizations.parent_org_id` self-FK + 깊이 정책 |
| 멤버 role 변경 | 기존 확장 | role PATCH API 신규(현재 미지원) |
| 소속 팀 직접관리 | 신규 테이블 | org↔team 관계 테이블 설계(큰 작업) |
| 회원 가입신청 | 신규 테이블 | 가입신청 모델 + 승인 워크플로 |
| 회비·정산 | 신규 테이블 + 결제 | 정산 모델(PartnerSettlement 참조) + 출금계좌 + PG 연동 — **결제=최대 승인 필요** |
| 공지·소통 | 신규 테이블 | org 공지/메시지 모델 + 알림 발송 |
| 인증·서류 | 신규 테이블 | 인증상태·서류제출·검토 모델 |
| 로고/배너 파일 업로드 | 인프라 | 스토리지(Vercel Blob/Supabase Storage) — 현재 URL 텍스트 저장뿐 |

- **운영 DB 단일 환경**: destructive 금지·schema 변경은 승인 선행(CLAUDE.md §DB 정책). 무중단 ADD(NULL 허용)라도 사용자 검토 후 적용.
