## ★ 설계 원칙 — 점진적 온보딩 (Progressive Onboarding) · 의뢰서 전반에 적용

> 사용자(수빈)가 정한 제품 철학. **이 의뢰서의 모든 화면 설계가 이 원칙을 따라야 합니다.** 디자인 클로드는 단체 생성·관리 화면을 그릴 때 이 톤을 1순위로 지킬 것.

**원칙 1 — 초기 진입은 무조건 간편하게.**
개별 유저 회원가입 / 팀 생성 / **단체 생성** 모두 **최소 정보로 즉시 완료**. 처음부터 서류·인증·계좌를 요구하면 사용자가 가입 자체를 꺼립니다. 첫 화면은 "장벽 0"이 목표.

**원칙 2 — 서류·인증은 "나중에", 실제로 필요해지는 시점에 유도.**
mybdr을 쓰면서 점진적으로 채워나가는 구조. 인증을 미뤄도 **기본 사용은 막지 않습니다**(강요 X, 유도 O). 필요한 순간에 부드럽게 안내.

> (참고) 회원가입·팀 생성도 동일 철학(간편 가입 → 점진적 인증)입니다. 단, 그 상세 화면은 이번 의뢰 범위 밖 — **이번 의뢰는 단체(organization) 중심**. 다만 디자인 클로드는 전체 제품이 같은 톤이라는 전제를 인지할 것.

### 단체에 적용되는 구체 흐름 (이 의뢰의 핵심 UX)

```
[즉시 생성]  최소 필드(이름·지역·소개)로 단체 생성 — 서류/인증/계좌 요구 0
     │
     ▼  (단체를 쓰다가 아래 시점에 도달하면)
[deferred 인증 유도]  실제로 필요해질 때 비로소 등록을 안내
     ├─ 정산·결제 문제 발생  (참가비 정산 / 출금 계좌 필요)
     ├─ 실제 대회 개최        (공식 대회로서 단체 효력 인증 필요)
     └─ 법적/공식 효력 증빙이 필요한 그 외 시점
```

이 흐름을 시각화하기 위해 디자인 클로드는 아래 3종 UI 패턴을 시안화합니다(상세 명세 = §4-5):
1. **단체 인증 상태 배지** (미인증/기본 → 서류 제출중 → 인증 완료) — 상세·관리 화면에 노출
2. **잠긴 기능 접근 시 단계적 안내 프롬프트/배너** ("이 기능을 쓰려면 OO 서류가 필요해요 → 지금 제출")
3. **단체 관리 화면 "인증·서류 제출 현황" 섹션** (체크리스트 — 무엇이 필요/완료인지)

> ⚠️ **백엔드 경계 (중요)**: 이 인증/서류/정산 연동 단계는 **현재 DB에 없는 신규 개념**입니다. 따라서 이 의뢰서에서는 **"시안·UX 컨셉 제안" 수준으로만** 다룹니다. 실제 DB 컬럼/테이블/결제 연동은 **백엔드 후속 설계 필요(별도 사용자 승인)**. → **시안은 만들되, 박제·DB는 후속.** 디자인 클로드는 이 영역을 §2 "실제 DB 컬럼" 규칙의 예외(=신규 컨셉)로 인지하고, 시안에 "컨셉(미구현)" 성격임을 자체 인지할 것.

## 0. 이 의뢰서의 핵심 — 디자인 시스템이 **두 개**다 (혼동 금지)

이 의뢰는 성격이 다른 두 영역을 다룹니다. **각 영역은 디자인 시스템과 적용 룰이 완전히 다릅니다.**

| 영역 | 대상 | 디자인 시스템 | 적용 룰 |
|------|------|--------------|--------|
| **Part 1. 공개 단체 사이트** | `(web)/organizations` | **BDR v2** (일반 사용자용) | ✅ 13룰 / AppNav frozen / Material Symbols / BDR 토큰(`var(--*)`) 전부 적용 |
| **Part 2. 단체 관리 콘솔** | `(admin-v2)/v2/ta/organizations` | **AdminShell** (운영자 콘솔) | ⚠️ **AppNav 없음 / Material Symbols 아님 / lucide 아이콘 / admin-v2 토큰** — 13룰 미적용 |

> ⚠️ **Part 2 주의**: admin-v2 콘솔은 BDR 13룰 대상이 **아닙니다**. 이미 박제된 `components/admin-v2`(AdminShell·PageHead·DataTable·Btn·Icon 등) + `styles/admin-v2` 토큰 + lucide 아이콘(kit.tsx `ICON_ALIAS`, kebab→Pascal)을 **그대로 재사용**합니다. 여기에 "Material Symbols / radius 4px / AppNav frozen"을 적용하면 정본 어긋남입니다. Part 2 시안은 기존 admin-v2 화면들(`/v2/ta/*`, `/v2/operate/*`, 백오피스 `/v2`)과 시각적으로 동일 계열이어야 합니다.

## 1. 컨텍스트

- **의뢰 대상**:
  - Part 1: `/organizations` (목록) + `/organizations/[slug]` (상세 — hero / 4탭)
  - Part 2: `/v2/ta/organizations` (단체 관리 콘솔 — 현재 읽기전용 stub)
- **우선순위**: P1 (사용자 요청 — "단체 생성·관리 + 단체 사이트 고도화")
- **박제 등급 변화**:
  - Part 1: B → A (데이터 빈약 상태 → "채워진 이상적 상태" 시각 완성)
  - Part 2: D(stub) → A (레거시 `tournament-admin/organizations` 6탭과 동등)
- **시급도**: 이번 batch (시안만 — 박제는 컷오버 조율 후)

### 1-1. 왜 이 작업인가 (현황 한 줄)

- 단체 도메인은 **백엔드 API·데이터 모델이 이미 완비**돼 있고, **레거시 admin 화면도 거의 완성**돼 있습니다.
- 그러나 ① **공개 사이트는 데이터가 비어 빈약해 보이고**(로고 0·배너 0·소개 2건), ② **그린필드 admin-v2 콘솔은 읽기전용 stub**(모든 버튼 "준비 중" 토스트)입니다.
- 따라서 이 의뢰는 **새 기능 설계가 아니라**, 이미 있는 백엔드·레거시 기능을 **(1) 채워진 이상적 상태의 공개 사이트 시안**과 **(2) admin-v2 콘솔 시안**으로 시각화하는 것입니다.

## 2. 데이터 기준 — 실제 DB 컬럼만 사용 (필수)

> ⚠️ **준비중(미존재) 필드는 시안에 절대 표시하지 마세요.** 아래는 실제 Prisma 스키마(`organizations` / `organization_members` / `tournament_series` / `Tournament`) 컬럼입니다. 이 목록 밖의 정보를 카드/필드로 그리면 박제 시 mock(허위 데이터)이 되어 reject됩니다.

### 2-1. ✅ 사용 가능 (실제 DB 컬럼)

**`organizations` (단체)**
| 필드 | 용도 |
|------|------|
| `name` | 단체명 |
| `slug` | URL (수정 불가 — 표시만) |
| `logo_url` | 로고 이미지 (현재 0건이나, 시안은 "채워진 상태" 전제) |
| `banner_url` | 배너 이미지 (hero 상단 배경) |
| `description` | 단체 소개 |
| `region` | 활동 지역 (서울/경기 등 17개) |
| `contact_email` / `contact_phone` | 연락처 |
| `website_url` | 외부 홈페이지 링크 |
| `is_public` | 공개 여부 토글 |
| `status` | pending / approved / rejected / archived (승인 워크플로) |
| `series_count` | 소속 시리즈 수 (집계 캐시) |
| `apply_note` | 신청 메모 (운영자 열람용·공개 미노출) |
| `approved_at` / `approved_by` / `rejection_reason` / `rejection_at` | 승인 워크플로 메타 |
| `created_at` | 가입/설립 표기는 "since 2024" 식 연도만 |

**`organization_members` (멤버)**: `role`(owner/admin/member 3종만), `is_active`, 연결 user(`nickname`·`email`·`profile_image_url`)
**`tournament_series` (시리즈)**: `name`·`slug`·`description`·`logo_url`·`status`·`tournaments_count`
**`Tournament` (회차/대회)**: `edition_number`·`venue_name`·`status`·`champion_team_id`(우승팀)·`startDate`

### 2-2. ❌ 미존재 — 시안에 그리지 말 것 (준비중)

| 그리지 말 것 | 이유 |
|------|------|
| 설립 연도(`founded_year`) | 컬럼 없음 — `created_at` 연도로만 "since YYYY" 표기 |
| 본부 주소(`address`) | 컬럼 없음 |
| 운영 원칙(`operating_principles`) | 컬럼 없음 |
| 주요 스폰서(`sponsors`) | 테이블·컬럼 없음 |
| 브랜드 색상(`brand_color`) | 컬럼 없음 — 색상은 `id` 해시 기반 자동 팔레트(`org-color.ts` pickColor 6색) |
| **소속 팀 목록/팀 수** | 단체↔팀 **직접 관계 없음** (시리즈→대회→팀만 존재) → Teams 탭은 **향후**(§5) |
| 직책 세분화(회장/부회장/총무) | role은 owner/admin/member 3종뿐 → "주최자/운영진/멤버"로만 표기 |
| 멤버 가입 신청(유저→단체) | 신청 모델 부재 — "가입 신청" 버튼은 **향후**(§5) |

> ✅ **예외 1건 — 인증/서류 컨셉은 의도적으로 시안화**: 위 "미존재 필드 표시 금지" 규칙의 **유일한 예외**는 §설계 원칙의 **점진적 온보딩 인증/서류 UX**(인증 상태 배지·잠금 안내·서류 현황 체크리스트)입니다. 이건 DB에 없지만 **"신규 UX 컨셉 제안"으로 일부러 그립니다**(§4-5). 다른 미존재 필드(설립연도·주소·스폰서 등)는 그대로 금지 — 둘을 혼동하지 말 것. 인증 영역만 "컨셉(미구현)" 성격으로 시안화.

## 3. Part 1 — 공개 단체 사이트 고도화 (O1 + O2)

> 디자인 시스템 = **BDR v2** (13룰·AppNav frozen·Material Symbols·BDR 토큰 전부 적용)
> 핵심 의도: **데이터가 가득 찬 이상적 상태**를 시안으로 보여줘 "단체 사이트가 매력적으로 보이는 목표 모습"을 확정. (현재 운영 데이터가 빈약하다는 사실과 무관하게, 로고·배너·소개·시리즈가 풍부히 채워진 상태를 그릴 것.)

### 3-1. 핵심 변경

1. **단체 상세 hero 강화 (O1)** — `banner_url`을 상단 배경으로, `logo_url`을 원형/라운드 로고로, `name`·`region`·`description`·회원수를 명확한 위계로. 배너/로고가 있을 때의 풍부한 hero 시안.
   - 배너 없을 때 폴백: `org-color.ts` 해시 그라디언트(현재 패턴 유지) — 단색/이니셜로 우아하게.
   - 로고 없을 때 폴백: 단체명 이니셜 박스.
2. **단체 목록 카드 강화 (O1)** — 로고·지역 태그·소개 한 줄·시리즈 수·회원 수가 채워진 카드 그리드. 로고 유무에 따른 두 상태 모두.
3. **Events 탭 = 단체-시리즈 연결 강조 (O2)** — "단체 → 시리즈 → 대회" 3단계 위계가 시각적으로 또렷하게. 시리즈 카드 안에 회차(대회) 리스트 + 상태 뱃지(모집중/진행중/종료) + 우승팀(`champion_team_id`). 시리즈가 여러 개 연결됐을 때의 풍성한 레이아웃.

### 3-2. 보존 (변경 금지)

- **AppNav frozen** (`03-appnav-frozen-component.md` 코드 그대로 — 메인 9탭/utility bar/5아이콘).
- 이미 실데이터로 동작하는 영역의 **데이터 구조·탭 키(`?tab=overview|teams|events|members`)** 유지 — 시각만 개선.
- **Members 탭**: 실데이터(owner/admin/member → 주최자/운영진/멤버). 직책 세분화 추가 금지.
- **Overview 탭**: `description`·`contact_email`·`contact_phone`·`website_url`만. 운영원칙/주소/스폰서 섹션은 **그리지 말 것**(준비중).

### 3-3. 레이아웃 / States

- 데스크톱 + 모바일(720px 분기) 두 폭. 카드 그리드는 `auto-fit minmax` 또는 className 분기(고정 grid 금지).
- States: 로고/배너 **있음 / 없음(폴백)** 두 상태를 반드시 함께 제시(현실 데이터가 비어 있으므로 폴백이 실제로 더 자주 보임).
- "가입 신청" 버튼은 현재 동작 미구현 → 시안에 **노출하지 말거나**(권장) 비활성(disabled) 상태로만.

### 3-4. Connections

- 진입: 더보기 "단체" 탭 / 홈 / 직접 URL.
- 목록 카드 클릭 → `/organizations/[slug]`.
- Events 탭 시리즈 클릭 → `/organizations/[slug]/series/[seriesSlug]` → 회차 클릭 → `/tournaments/[id]`.

## 4. Part 2 — 단체 생성·관리 콘솔 (admin-v2 그린필드)

> 디자인 시스템 = **AdminShell** (`components/admin-v2` + `styles/admin-v2` 토큰 + lucide 아이콘). **AppNav·Material Symbols·13룰 미적용.**
> 핵심 의도: 현재 읽기전용 stub인 `/v2/ta/organizations`를, 레거시 `tournament-admin/organizations`(6탭·완성)와 **기능적으로 동등한** 콘솔 화면으로. 기존 admin-v2 화면(`/v2/ta/*`, `/v2/operate/[id]`)과 같은 셸·같은 컴포넌트 계열.

### 4-1. 화면 ① 단체 목록 (개선)

- 현재: `ad-cardgrid` 카드(로고/인증뱃지/대회·운영진·회원 통계/관리·운영진 버튼) — **읽기는 OK**.
- 시안 요구: "단체 등록" 버튼이 실제 폼으로 이어지는 흐름 + 카드 "관리"/"운영진" 버튼이 상세로 이어지는 흐름(현재는 토스트). Empty 상태(정본 `Empty` 컴포넌트) 유지.

### 4-2. 화면 ② 단체 생성 폼 (신규) — **점진적 온보딩 1순위 적용**

> ★ §설계 원칙 직접 적용 화면. **"장벽 0, 즉시 생성"이 이 화면의 최우선 목표.**

- **최소 필드로 즉시 생성**: `name`(필수) / `region`(드롭다운) / `description` 정도면 충분. `slug`(자동생성)·`logo_url`·`contact_email`·`website_url`은 **선택(나중에 채울 수 있음 톤)**.
- ⛔ **생성 단계에서 요구 금지**: 사업자 서류 / 대표자 인증 / 계좌 / 결제 정보. 이런 건 생성 폼에 **칸 자체를 두지 말 것**. (요구하면 가입을 꺼린다는 게 핵심 철학)
- 폼 하단/완료 화면에 **"서류·인증은 나중에, 필요할 때 안내해 드려요" 안내 문구** 1줄(강요 X·안심 톤).
- admin-v2 폼 컴포넌트(입력·라벨·드롭다운·제출 버튼) 정본 패턴 사용. 제출 후 → 상세(화면 ③)로 이동.
- (선택 필드는 "더 입력하기" 접기/펼치기로 숨겨 첫인상을 가볍게 하는 것도 권장.)

### 4-3. 화면 ③ 단체 상세 — 6탭 (신규, 레거시 동등)

레거시 `(admin)/tournament-admin/organizations/[orgId]` 6탭 구조를 admin-v2 셸로 재현:

| 탭 | 내용 | 비고 |
|----|------|------|
| **기본 정보** | 조회 + **편집 모달/패널** | 편집 필드 = name·description·region·contact_email·**contact_phone**·website_url + **로고/배너 편집 UI**(아래 4-4) + is_public 토글 |
| **멤버** | 멤버 목록(주최자/운영진/멤버 뱃지) + 초대(이메일·role 선택) + 제거(confirm) | role 변경 토글은 **표시만**(백엔드 미지원 — 동작 없음, 준비중 처리) |
| **시리즈** | 시리즈 목록 + 새 시리즈 추가 + "기존 대회 가져오기"·"이동/분리" 액션 | 레거시 AbsorbTournamentsModal·MoveSeriesModal 흐름 시각화 |
| **회차** | 모든 시리즈의 회차(대회) 통합 표(name·status·start_date·venue·max_teams) | 회차 클릭 → 대회 운영 |
| **운영진(권한 위임)** | admin role 멤버 목록 + 권한 안내 | **UI만**(역할 변경 동작 미지원 — 준비중 안내 박스) |
| **활동 이력** | 빈 상태(준비중) | ORG_ACTIVITY_LOG 테이블 없음 → 정본 Empty + "준비 중" |

### 4-4. 화면 ③ 강조 — 로고/배너 **편집 UI** (레거시에도 없는 갭)

- 백엔드 PATCH API는 `logo_url`·`banner_url` 수정을 **이미 지원**하지만, 레거시·admin-v2 모두 편집 UI가 없음 → 이 의뢰의 핵심 신규 시안.
- 요구: 기본 정보 탭(또는 별도 "브랜딩" 섹션)에서 로고/배너를 미리보기 + 변경할 수 있는 UI.
- ✅ **[PM 결정 2026-06-29] 이미지 업로드 위젯으로 시안화**: 로고/배너는 **드래그&드롭 업로드 위젯(드롭존 + 미리보기 + 교체/삭제)** UI로 그립니다. (URL 텍스트 입력 방식 아님.)
  - 업로드 위젯 UI: 드롭존 영역 / 파일 선택 버튼 / 업로드 후 썸네일 미리보기 / 교체·삭제 액션 / 권장 사이즈·용량 안내 문구. admin-v2 정본 컴포넌트 톤 유지.
  - ⚠️ **백엔드 경계**: 현재 백엔드는 `*_url` 텍스트 저장 방식이고, **실제 파일 업로드 인프라(스토리지: Vercel Blob/Supabase Storage 등)는 별도 확인·구축이 필요**합니다. → **시안(업로드 위젯 UI)은 제작하되, 실제 업로드 동작·스토리지 연동은 백엔드 후속(§5 O7 계열, 별도 사용자 승인).** 디자인 클로드는 업로드 위젯을 "목표 UI 컨셉"으로 그리고, 와이어링은 후속임을 인지할 것.

### 4-5. 화면 ③ 핵심 신규 — **점진적 온보딩: 인증/서류 deferred UX 컨셉** (★ 이번 의뢰 하이라이트)

> ★ §설계 원칙의 "deferred 인증 유도"를 시각화하는 UI. **DB에 없는 신규 UX 컨셉 — "컨셉(미구현)" 성격으로 시안화**(§2-2 예외). 박제·DB·결제 연동은 백엔드 후속(별도 사용자 승인). 디자인 클로드는 "이건 목표 UX를 보여주는 컨셉 시안"임을 인지하고 그릴 것.

시안화할 3종 패턴:

**① 단체 인증 상태 배지** (상태 머신 시각화)
- 단계: `미인증/기본` → `서류 제출중(검토 대기)` → `인증 완료` (필요 시 `반려/보완 필요` 추가).
- 노출 위치: 단체 상세 헤더 + 관리 콘솔 카드/상세 상단. admin-v2 배지/칩 정본 컴포넌트(`Icon`+tone) 사용.
- 톤: 미인증이라도 **부정적/경고가 아닌 중립**("기본 단체" 정도) — 인증은 보너스라는 인상.

**② 잠긴 기능 접근 시 단계적 안내 프롬프트/배너**
- 트리거 시점(§설계 원칙 흐름): **정산/출금 계좌 필요** 시, **실제 대회 개최(공식 효력)** 시.
- UI: 잠긴 기능(정산 탭·공식대회 개최 버튼) 위/옆에 자물쇠 + 안내 배너. 문구 예: "참가비 정산을 받으려면 출금 계좌·사업자 정보가 필요해요 → 지금 등록하기".
- **기본 사용은 막지 않음**: 잠긴 건 정산·공식효력 같은 "실질 결제/법적" 기능에 한정. 단체 생성·운영·시리즈 관리 등은 인증 없이 그대로 사용.
- 액션 버튼은 "지금 등록"(유도) + "나중에"(닫기) 2지선다 — 강요 X.

**③ 단체 관리 화면 "인증·서류 제출 현황" 섹션** (체크리스트)
- 관리 콘솔 상세에 별도 섹션 또는 탭 카드. **무엇이 필요/완료인지 체크리스트** 형태.
- 예시 항목(컨셉): 대표자 인증 / 사업자(또는 단체) 증빙 / 출금 계좌 / 공식대회 자격. 각 항목 = 미완료/검토중/완료 상태 + "제출하기" CTA.
- 미완료여도 진행률만 보여주고 잠그지 않음(진행률 N/M). "지금 다 안 해도 괜찮아요" 톤.

> 백엔드 경계 재확인: 위 배지/잠금/체크리스트가 실제로 동작하려면 신규 모델(예: 인증 상태·서류 제출·계좌·결제 연동)이 필요 → **시안만, 박제·DB는 후속 설계 + 사용자 승인.** 시안에는 샘플 상태(미인증/제출중/완료)를 컨셉 데이터로 표현하되, "실데이터 와이어 가정 금지".

### 4-6. 보존 / States (Part 2)

- 기존 admin-v2 정본 컴포넌트(PageHead·DataTable·Btn·Icon·Modal·Empty)·토큰·lucide 아이콘 재사용. 새 색/새 셸 만들지 말 것.
- States: empty(단체 0건)·loading·success(저장 토스트)·error·disabled(준비중 버튼).
- 슈퍼관리자 승인 큐(`/admin/organizations`)는 이미 리스킨됨 → **이번 범위 제외**(참고만).
