역할: 너는 MyBDR(전국 농구 매칭 플랫폼)의 시니어 프론트엔드 엔지니어다.

목표: 첨부한 `design_handoff_admin/` 패키지의 시안과 **디자인·기능·상태·데이터까지 1:1로 동일한** 관리자 영역을 구현한다. 기존(레거시) 관리자 코드는 **전량 제거**하고 새로 만든다.

대상 코드베이스: <타깃 저장소/경로 — 예: github.com/org/mybdr-web, 앱은 Next.js + TypeScript + …>
(코드베이스가 없으면 가장 적합한 스택을 골라 새로 구성하고 그 이유를 1줄로 보고할 것.)

반드시 먼저 읽을 것 (순서대로):
1. `START-HERE.md`        — 정본 선언 · 실행법 · 토큰 · 진입점 연결 맵 · 레거시 제거 체크리스트
2. `screenshots/INDEX.md` + `screenshots/*.png` — 화면별 시각 정본(결과물은 이 캡처와 동일해야 함)
3. `PARITY-MATRIX.md`     — 시안 컴포넌트/class/function → 구현 파일 1:1 매핑
4. `CLEANUP-MANIFEST.md`  — 레거시 삭제·교체 기준 + 아이콘 매핑
5. `DATA-CONTRACT.md`     — 화면별 필드 + 🔴 신규 필드 9건
6. `TOURNAMENT-OPS-STATES.md` · `ADMIN-TOSS-STATE-QA.md` · `RESPONSIVE-QA.md` — 상태/반응형 사양

핵심 규칙 (위반 금지):
- 번들의 .html/.jsx는 **프로토타입 레퍼런스**다. 그대로 배포하지 말고, 타깃 스택의 컴포넌트/라우팅 패턴으로 **재현**한다. 단 시각/동작 결과는 시안과 동일해야 한다.
- 관리자 영역 디자인시스템 = **Toss**(`toss.css` 토큰: --primary #3182F6, 라운딩 card24/btn16/input14/chip12, Pretendard + JetBrains Mono). 아이콘은 **lucide만**(Material Symbols 금지).
- 관리자 영역은 사용자 공개 앱(BDR Red/다크/Material 시스템)과 **완전 분리**한다. 두 시스템을 섞지 말 것.
- 토큰 값(색·간격·라운딩·그림자·타이포)은 시안 그대로. 임의 변경 금지.
- 모든 서버 호출은 시안에서 mock이다. 클라이언트 동작(필터·토글·캘린더·드래그·import 파싱·토스트·드로어)은 **실제 동작**으로 구현하고, 데이터만 실제 API/모델로 치환한다. `DATA-CONTRACT.md`의 신규 필드 9건을 모델에 반영한다.

구현 범위 (9 진입 화면):
- 관리자 홈 / 대회 운영 워크스페이스(6메뉴·7패널) / 대회 생성(5단계) / 대회 수정 / 대회 관리자 셸 / 백오피스(18화면) / 협력업체 콘솔 / 심판 관리자(12화면) / 공개 토너먼트 사이트.
- 화면 간 네비게이션은 `START-HERE.md` §9 "진입점 연결 맵" 그대로 배선한다(예: 백오피스 대회 행 → 대회 운영 워크스페이스, 대회 관리자 목록 행 → 대회 운영, 콘솔 브랜드 → 관리자 홈 등). 데드버튼 0 보장.
- 공개 사이트는 `public-site-*`(44팀/27경기 통일본) 기준. 구 `토너먼트 사이트.html`(38팀)은 대체한다.

작업 방식:
1. 먼저 위 문서를 읽고 **구현 계획**(파일/라우트 구조, 공유 컴포넌트 목록, 레거시 제거 목록, 데이터 모델 변경, 단계 분할)을 제시하고 승인을 받는다.
2. 공유 기반부터 구현: 토큰 → 공통 컴포넌트(Btn/Badge/Modal/Toggle/Check/StepDots/Empty/AdminShell/PageHead/KpiGrid/DataTable/SchemaList) → 화면.
3. 각 화면 완료 시 해당 스크린샷과 나란히 비교해 픽셀 정합을 자가 점검한다.
4. 상태(loading/empty/error/saving/saved/permission/destructive)와 반응형(390/720/1024/1440)을 QA 체크리스트로 통과시킨다.
5. PR 단위는 "공유 기반" → "콘솔별" → "워크스페이스" → "공개 사이트"로 나눈다.

산출 보고: 각 단계마다 (a)구현 파일, (b)시안 대비 차이/미결, (c)신규 필드/엔드포인트 요청을 정리해 보고한다.
