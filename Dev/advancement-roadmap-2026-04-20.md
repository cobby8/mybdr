## 2. 이 속도를 만든 5가지 원리 (반복 가능성을 위해 고정)

이틀 만에 4주치를 소화한 건 운이 아니고 **작업 구조** 덕입니다. 이 구조를 명시적으로 남겨두면 앞으로도 반복 가능합니다.

1. **프롬프트 묶음이 "파일 경로·요구사항·디자인 스펙·검증 방법" 4요소를 항상 포함**
   - `ux-implementation-plan-2026-04-19.md` 같은 플랜이 Claude Code 복붙용으로 바로 쓰였기 때문에 PM 역할 부담이 최소화됨
   - **교훈**: 다음 트랙(L2/L3/운영 DB) 착수 전에도 **플랜을 "프롬프트화"** 해서 저장. 지금까지처럼 `Dev/` 아래 `.md` 하나로 유지

2. **묶음 PR (#45/#46/#47/#49) — 하나의 PR에 여러 묶음 소화**
   - 1인 작업이라 충돌 없음 → PR 수를 줄이고 하루 끝 1~2 PR 묶음이 효율
   - **교훈**: 원영과의 공동 작업 영역이 아니라면 계속 이 스타일. 단 dev 누적은 하루 안에 털기

3. **세션 분리** — 본 세션(UX) vs 카페 sync 세션이 별도 터미널에서 동시 진행
   - 4-20 scratchpad "세션 분리 원칙" 문서화 완료
   - **교훈**: 다음 트랙도 성격이 다르면(예: L3 = IA 시각 / 운영 DB = 인프라) 세션을 분리해 컨텍스트 오염 방지

4. **knowledge 6파일이 "재발 방지 메모리"** 로 작동
   - apiSuccess 재발 5회 가드, 공식 기록 쿼리 가드, 쿠키 갱신 절차 등이 모두 `.claude/knowledge/` 에 쌓여 같은 실수를 반복 안 함
   - **교훈**: 30분+ 삽질 or 2회+ 재발하면 즉시 knowledge에 기록하는 루틴 유지

5. **매일 종료 루틴 고정** — tsc → 커밋 → push → scratchpad 1줄 → knowledge 검토
   - 이 루틴이 다음날 시작 부담을 "5~10분의 점검"으로 압축
   - **교훈**: 바쁘더라도 "오늘 작업 마무리" 프롬프트 1회는 생략 금지

## 5. 매주 유지 가능한 페이스 공식

이틀에 4주치를 소화한 페이스는 **지속 가능하지 않음**. 다음 공식으로 지속 가능한 버전을 고정하기를 권장합니다.

```
주 5일 × 하루 4시간 + 저녁 선택 1~2시간 = 20~30시간/주
4주 = 80~120시간 = "분기 큰 덩어리 1개 + 중간 기능 2개 + 부채 정비"
```

분기 단위 큰 덩어리 예시 (1개만 고르기):
- L2 프로필 통합 (~15h)
- 운영 DB 분리 (~9h)
- 트랙 A 하이라이트 자동화 (~12h)

중간 기능 (2개):
- 트랙 B 중 2개
- 또는 트랙 C 중 2개

부채 정비:
- 하드코딩 색상 5~10개/주
- any 1~2개/주
- knowledge consolidate 월 1회

### 위험 신호

이런 신호가 보이면 속도를 낮춰야 합니다:
- **knowledge 갱신이 줄어듦** — 실수가 쌓이기 시작한 신호
- **PR 본문이 짧아짐** — 변경을 설명할 여력이 없다는 신호
- **`tsc` 에러를 "나중에"** 로 미룸 — 기반이 흔들리는 신호
- **새 기능이 재미 없게 느껴짐** — 번아웃 초기 신호
