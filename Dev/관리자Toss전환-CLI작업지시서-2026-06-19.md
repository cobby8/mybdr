## 🚨 0. 가장 중요 — 디자인 분기 (절대 혼동 금지)

mybdr에는 이제 **두 개의 독립 디자인시스템**이 공존한다. 화면이 어느 쪽인지 먼저 판별하고 그 시스템만 적용한다.

| 영역 | 디자인시스템 | 토큰/규칙 |
|------|------------|----------|
| **사용자 프론트** `(web)/*` (홈/경기/대회 공개/팀/코트/랭킹/커뮤니티/프로필 등) | **mybdr 기존** | var(--color-*)·**다크 기본**·4px·**Material Symbols**·lucide 금지·BDR Red #E31B23. 기존 13룰·AppNav frozen 그대로. **건드리지 말 것.** |
| **관리자/운영** `(admin)/admin/*` · `(admin)/tournament-admin/*` · `(web)/partner-admin/*` · `(referee)/referee/*` | **Toss (신규)** | `toss.css` 토큰·**라이트 전용**·24/16/14/12px·**lucide**·Toss Blue #3182F6. AppNav·다크·Material **미적용**. |
| **참가신청서**(대회 참가 신청 흐름) | **Toss (예외)** | 사용자가 쓰지만 대회 모듈이라 **Toss 적용**(시안 결정). 라우트 위치는 기존 대회 참가 신청 경로 유지. ⚠️ 이 한 화면만 "사용자용인데 Toss" — 헷갈리지 말 것. |

**금지 사항**:
- 관리자 화면에 mybdr var토큰/다크/Material/AppNav 적용 ❌ → Toss(toss.css)만.
- 사용자 프론트에 Toss/lucide/라이트 침범 ❌.
- lucide는 **관리자 영역 한정 허용**. 사용자 프론트는 여전히 lucide 금지.
- 시안 toss-kit.jsx의 `window.lucide.createIcons()` CDN 주입은 **프로토타입 방식** → 운영은 **`lucide-react`(npm) 컴포넌트**로 이식. CDN injection 베끼지 말 것.

## 1. 두 트랙 — 리스킨 vs 리빌딩 (작업 성격 구분)

| 트랙 | 대상 | 작업 성격 |
|------|------|----------|
| **A. 리스킨 (비주얼만)** | `(admin)/admin/*` 백오피스 전부(**대회관리 제외**) + `partner-admin/*` + `referee/*` | **기능·컬럼·필터·정렬·액션·문구·라우트 1:1 유지.** 비주얼만 Toss로 교체. |
| **B. 리빌딩 (기능 변경)** | **대회관리만** = `(admin)/admin/tournaments` + `tournament-admin/*` | 종별 마스터·참가신청·대회상세 IA까지 새로 설계(DATA-BINDING 계약). |

### 🔒 트랙 A 절대 규칙 (위반 = 회귀)
1. **레이아웃·정보위계·기능(필터/정렬/액션/모달)·문구 변경 금지.** 비주얼만 교체.
2. **컬럼 추가/삭제 금지, 라우트 신설 금지.** 기존 mybdr admin 화면이 기능 SoT.
3. 시안 `admin-data*.jsx`의 mock 컬럼은 **참고용** — 실제는 기존 mybdr admin route.ts/페이지의 컬럼·필터를 그대로. mock과 실제가 다르면 **실제 우선**.
4. 매핑: 카드→`.ts-card` / 버튼→`.ts-btn` / 테이블→`.ts-table` / 상태칩→`.ts-badge` / 입력→`.ts-input` / Material 아이콘→lucide 동의어.

## 2. 디자인시스템 이식 (Phase 0 — 1순위 기반물, 전 화면 선행)

1. `toss.css` `:root` 토큰을 mybdr 관리자 공통 CSS/Tailwind 레이어로 이식 — **사용자 프론트 토큰과 네임스페이스 분리**(예: `.admin-shell` 스코프 또는 `data-skin="toss"` 하위). 프론트 var(--color-*)와 섞이지 않게.
   - 토큰: `--bg #F2F4F6 / --card #fff / --ink #191F28 / --primary #3182F6 / --ok #15B86A / --warn #FF9500(대기접수) / --danger #F04452` + Grey 50~900 + radius(card24/btn16/input14/chip12) + shadow xs~lg. Pretendard.
2. `toss-kit.jsx`(Icon/Btn/Badge/Toggle/Check/StepDots/Modal/Empty) + `toss-admin.jsx`(StatusTabs/FilterBar/DataTable/DetailModal/StatCard/PanelRow)를 **운영 React 컴포넌트로 이식**. lucide는 lucide-react.
3. 이후 모든 관리자 화면이 이 키트만 사용.

## 4. 트랙 B / P0 데이터 계약 (DATA-BINDING.md 준수)

### 4-1. 종별 마스터 = `admin_categories` (잃어버린 "종별 프리셋"의 실체)
- 신규 테이블 `admin_categories { id, name, divisions string[], ages string[] }`.
- **4종 시드(사용자 실데이터 — 정확히)**: 일반부(D3 D4 D5 D6 D7 D8) / 유청소년(하모니 i1 i2 i3 i4 · 연령 U8~U18) / 대학부(U1 U2 U3) / 시니어(S1 S2 S3 · 연령 +40 +45 +50 +55 +60 +65 +70).
- ⚠️ BDR-join-v1의 `tournament_presets`/PresetManager는 **미연결 고아** → **베끼지 말 것**.
- CRUD: divisions·ages 태그 입력(Enter 추가/X 제거).

### 4-2. 참가신청 = **팀 로스터 조인** (명단 직접입력 폐지 — 핵심)
- 흐름: 로그인 사용자 가입 팀 조회 → 팀 선택(1개면 자동)·정보 확인 → 종별·디비전(div_caps+집계로 WAITING 판정) → **팀 roster(기존 players)에서 출전선수 체크 선택** → 제출 시 서버 정원 재판정 status APPLIED|WAITING.
- 유니폼 입력 없음(팀 설정값 표시만). 비번 없음(세션).
- 신규 조인: 출전선수 선택 저장(예: `tournament_team_players`).
- ⏳ **출전 최소인원·게스트 추가 = 추후 결정** → `MIN_PLAYERS_GUARD`/`ALLOW_GUEST` 토글 자리만(off).

### 4-3. 참가팀 상태 머신
- `status`: APPLIED|WAITING|CONFIRMED|CANCELED / `payment_status`: pending|paid|refunded.
- **입금완료(paid)→자동 CONFIRMED**. mybdr `TournamentTeam.status`(pending/approved/rejected/paid) 매핑.

### 4-4. 종별 확정 자동화
- `DivisionGeneratorModal`이 admin_categories 읽어 종별 생성 → `TournamentDivisionRule` 자동 생성(code/label/format/settings/정원/참가비). 이전 대회 복사(format/settings 포함).
